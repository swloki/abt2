🎯 核心问题

一个 ERP 系统的本质是：很多人同时对同一批业务单据做操作，系统必须保证这些操作不会把数据搞乱。

比如一张销售订单：

  • 张三点了「确认」
  • 李四点了「发货」
  • 王五想点「取消」

这三个人不能同时成功，也不能乱序——你不能先发货再确认。谁来保证这个顺序是对的？这就是系统核心基础设施要解决的根本问题。

答案分成三个层次：

状态机
这个操作合法吗？
→
工作流引擎
按什么步骤走？
→
EventBus
谁需要知道？
🔒 一、状态机 — "能不能这么做"

没有状态机时会怎样

✗ 每个 方法自己判断
async fn confirm(&self, ...) {
    if order.status != Draft {
        return Err("只有 Draft 可以确认");
    }
    // ... 业务逻辑
}

async fn complete(&self, ...) {
    if order.status != Shipped {
        return Err("只有 Shipped 可以完成");
    }
    // ... 业务逻辑
}

状态转换规则散落在每个方法里。10 个状态、20 个转换时，规则变成没人能看到的"潜规则"。

✓ 集中到状态机统一管理
// 状态机的全部核心逻辑
fn transition(entity_type, entity_id, to_state) {
    let current = get_current_state(...);
    let rule = find_rule(from: current, to: to_state);

    if rule.is_none() {
        return Err("非法转换");
    }

    write_log(current → to_state);  // 不可变审计日志
    execute_side_effects(rule);   // 触发事件/通知
}

规则集中在数据库，全貌一目了然，改规则不需要改代码。

三层架构

第 1 层：数据库定义 — state_transition_defs 表
存储 (entity_type, from_state, to_state) 三元组，定义"什么是合法的转换"
▼
第 2 层：StateMachineService — 验证 + 日志
查规则 → 验证 → 写日志 → 执行 SideEffect
▼
第 3 层：业务 Service — confirm() / complete() / cancel()
先调状态机验证，再执行库存/价格等业务操作
状态机回答的只是一个极其简单的问题："从状态 A 到状态 B，这条边在状态图上存不存在？"它不关心为什么合法，不关心发货了多少数量。它只做判断。

SideEffect — 转换不只是"改个字段"

SideEffect 是挂在状态转换规则上的"附加动作"，存储在 state_transition_defs.side_effects JSONB 列中。

pub enum SideEffect {
    PublishEvent { event_type, payload_template },  // ✅ 已实现 — 往 EventBus 写事件
    Notify { role_ids, template },                  // 🔲 预留 — 发通知
    TriggerWorkflow { definition_id },            // 🔲 预留 — 触发工作流
    UpdateField { field, value_template },         // 🔲 预留 — 自动更新字段
}

执行逻辑在 StateMachineServiceImpl::transition() 的第 4 步——遍历 SideEffect 列表，match 分发。只有 PublishEvent 已实现，其他三个是预留接口。

SideEffect 不是 EventBus 的执行者，而是 EventBus 的调用者。它是状态机的特性，不是 EventBus 的。PublishEvent 变体恰好会调用 EventBus.publish()。
📡 二、EventBus — "做了之后谁要知道"

没有 EventBus 时会怎样

✗ 直接调用下游模块
// 销售订单确认时，直接调用库存和采购
async fn confirm(&self, ...) {
    inventory_service.reserve(...);     // ← 直接依赖
    purchase_service.check_replenishment(...); // ← 直接依赖
    h3yun_client.sync_product(...);       // ← 直接依赖
}

每加一个下游就要改这个文件。销售模块不需要知道库存怎么预留、采购怎么补货。

✓ 公告栏解耦
// 销售只说："订单确认了，爱谁谁"
event_bus.publish(SalesOrderConfirmed, order_id);

// 下游各自响应
库存模块: reserve(...)
采购模块: check_replenishment(...)
外部系统: sync(...)

发布者不需要知道谁在听，听的人不需要知道谁发的。只约定事件类型作为共同语言。

Outbox 模式 — 保证不丢消息

业务操作（如确认订单）
│
① 开数据库事务
② 更新订单状态（业务数据）
③ 写入 domain_events 表（事件）
④ 提交事务
同一个事务 →
✓ 事务成功 → 事件一定写入了
✓ 事务回滚 → 事件也没了
不会出现假消息
▼
后台 EventProcessor
PostgreSQL LISTEN/NOTIFY 实时通知 → 取出 Pending 事件 → 分发给 EventHandler → 成功 mark_processed / 失败重试，超限进死信队列
Outbox 的核心洞察：把事件和业务数据写进同一个数据库事务，保证它们要么一起成功、要么一起失败。然后用后台进程把事件搬运出去。

三大保证

保证怎么实现的
不丢Outbox 模式（同事务）；Processor 断线重连；孤儿事件自动恢复
不重复idempotency_key 唯一约束（ON CONFLICT DO NOTHING）；消费端也要幂等
可追踪trace_id / request_id 贯穿全链路；entity_state_logs 记录每次状态变化
🔀 三、工作流引擎 — "按什么步骤走"

与状态机 + SideEffect 的区别

三者都在回答"某件事发生后，下一步该做什么"，但复杂度完全不同：

SideEffect
同步、单步、无分支、无等待
"发个事件"
→
状态机转换
同步、有规则校验
"A→B 合法吗"
→
工作流引擎
异步、有分支、有等待
"走完这个流程图"

对比详情

状态机 SideEffect工作流引擎
执行方式 同步 同一个事务里 异步 可跨天跨人
复杂度 一条转换 → 几个动作 图遍历，有分支 / 汇合 / 条件
等待 没有，做完就返回 有，审批节点等人操作
分支 没有 有，条件决定走哪条边
典型场景 状态变了发个事件 采购单需要三级审批
代码量 一个 for + match 递归图遍历 process_next_node()

工作流的执行模型

process_next_node(tx, instance_id, graph, node_id, condition) {
    // 1. 先判断边上的条件
    if !evaluate_condition(condition) { return; }

    // 2. 根据节点类型做不同的事
    match node.node_type {
        Approval  => create_task(assignee),   // 创建待办，等真人审批，return
        AutoTask  => process_next_node(...next), // 递归往下走
        Join      => if all_branches_done {       // 等所有分支汇合
                        process_next_node(...next)
                     },
        End       => mark_completed(),
    }
}
状态机是"门卫"（能不能做），SideEffect 是"门铃"（做了之后通知谁），工作流是"完整的流程编排"（按步骤一步步走）。SideEffect::TriggerWorkflow 是串联两者的桥梁。
🔗 四、三者的关系 — 层级接力
① 状态机
Draft → Confirmed 合法吗？ ✓
├── 写审计日志
└── SideEffect::TriggerWorkflow
▼
② 工作流引擎
Start → 部门经理审批 → 财务审批 → End
├── （等了 2 小时）部门经理 approve
├── （等了 1 天）财务 approve
└── 流程完成
▼
③ EventBus
广播"审批通过"事件
├── 采购模块 → 通知供应商
├── 库存模块 → 生成入库计划
└── ...其他下游模块各自响应
状态机是起点，工作流是中间编排，EventBus 是终点广播。三者通过 SideEffect 和事件类型串联成一条链。每一层只管自己的事。
层职责不关心
状态机这个转换合法吗审批要走几步、谁来审
工作流按流程图一步步走业务数据是什么
EventBus谁关心这件事就通知谁状态怎么变、流程怎么走
🦴 五、完整骨架 — 10 个核心组件

状态机 / 工作流 / EventBus 管的是"流程和通信"。但一个 ERP 系统还有几件同样根本的事要做。abt-core/src/shared/ 共 14 个共享服务，按第一性原理分成三组：

你已经理解的：状态机 → 工作流 → EventBus
身份与权限
"你是谁，能做什么"
Identity
数据可追溯
"发生了什么，能查吗"
AuditLog · DocSequence · DocLink · StateLog
业务能力
"具体业务怎么算"
InventoryResv · CostEntry · Idempotency

一句话定位表

组件回答的根本问题一句话
状态机 这个操作合法吗 门卫
工作流 按什么步骤走 流程编排
EventBus 谁需要知道 广播通知
Identity 你是谁，能做什么 门禁
Audit Log 发生了什么变更 监控录像
Doc Sequence 单据编号唯一吗 发号器
Doc Link 单据之间什么关系 血缘图谱
Inventory Reservation 还能卖多少 库存看门人
Cost Entry 钱怎么走的 账房先生
Idempotency 会不会重复 去重过滤器

各组件详细说明

Identity（身份认证 + 权限）
每个请求进来第一件事：你是谁？有权限吗？
#[require_permission("PRODUCT", "delete")]
Audit Log（审计日志）
记录字段级 diff："张三把金额从 10000 改成 12000"
不可篡改，出了问题能追溯。
Document Sequence（单据编号）
SO-2026-06-00142，唯一、连续、不可跳号。
ERP 财务合规的硬性要求。
Document Link（单据关联）
报价单 → 订单 → 发货单，整条链可追溯。
任意一张单据都能找到上下游。
Inventory Reservation（库存预留）
多个订单抢同一批库存时，谁先确认谁先得。
回答 ATP：还能卖多少？
Cost Entry（成本分录）
复式记账——每笔业务产生成对会计分录，借贷必相等。
ERP 财务核心。
Idempotency（幂等性）
用户点了两次"提交"，不会创建两张订单。ON CONFLICT DO NOTHING 防重复。
🎬 六、完整场景 — "确认销售订单"

用户点击"确认销售订单"，10 个组件如何依次工作：

Identity 张三有权限吗？ ✓
StateMachine Draft → Confirmed 合法吗？ ✓
StateLog 写状态变更日志 entity_state_logs
SideEffect 发事件 SalesOrderConfirmed
AuditLog 记录所有字段变更（金额、状态等）
DocSequence 订单号 SO-2026-06-00142 已生成
InventoryReservation 预留库存，计算 ATP
CostEntry 生成成本分录（借贷）
EventBus 广播事件，下游模块各自响应
Workflow （如果配置了 TriggerWorkflow）启动审批流程
状态机 / 工作流 / EventBus 是"流程中枢"，但还需要身份、追溯、业务计算这三组能力才能组成一个完整的 ERP。
