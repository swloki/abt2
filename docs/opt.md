# 工序工资结算系统设计方案

> 场景：LED 模组 & 电源小工厂，不同产品工序不同，按件计价，手机扫码
> 技术路线：独立轻量 Web 应用（H5）+ API 对接自研 ERP
> 参考：Foxconn / BYD / 电子代工厂 MES 工序管理最佳实践
> 核心目标：扫码报工 → 自动算工资 → 物料损耗对比

---

## 一、业务流程总览

### 原方案 vs 优化后

```
原方案（有问题）：                          优化后（大厂标准流程）：

领料（ERP出库单）                          创建生产工单（指定产品+数量）
    ↓                                          ↓
生成生产批次 ← 一个出库单怎么拆？            ERP 自动生成领料单（根据 BOM）
    ↓                                          ↓
员工扫码开工                               仓库发料 → 生成生产批次
    ↓                                          ↓
组长确认数量                               每个批次 = 一张【工序流转卡】
    ↓                                          ↓
下一道工序                                 ┌→ 员工扫流转卡开工
    ↓                                      │      ↓
入库                                      │   报工（良品数+不良数+原因）
    ↓                                      │      ↓
算工资                                    │   下一道工序（扫同一张流转卡）
                                             │      ↓
                                             └── 最后工序 → 质检 → 入库
                                                    ↓
                                               月末自动算工资 → 同步 ERP
```

### 核心变化

| 变化点 | 原方案 | 优化后 | 为什么改 |
|--------|--------|--------|----------|
| 流程起点 | 领料出库单 | **生产工单** | 大厂标准：先有工单再有领料，工单是生产指令 |
| 批次生成 | 不清楚怎么拆 | **工单拆批次，组长操作** | 一个工单按班组/日期拆成多个批次 |
| 身份载体 | 扫工单号 | **工序流转卡**（QR 码标签） | 流转卡跟着物料走，每道工序扫同一张卡 |
| 报工数据 | 只报数量 | **良品数 + 不良数 + 不良原因** | 区分谁的责任，影响工资计算 |
| 工资计算 | 数量 × 单价 | **良品数 × 单价**（不良品不计） | 大厂通例：只有合格品才计工资 |

## 二、核心概念解释

### 1. 生产工单（Work Order）— 生产指令

生产工单是整个流程的起点，由管理员（或 ERP）创建。

**包含信息：** 产品型号、计划生产数量、计划完成日期、优先级

**示例：**

| 工单号 | 产品 | 计划数量 | 计划完成日期 | 状态 |
|--------|------|----------|-------------|------|
| WO-20260407-001 | LED 模组 A | 2000 | 4/10 | 进行中 |
| WO-20260407-002 | 电源 C | 500 | 4/8 | 进行中 |

### 2. 生产批次（Production Batch）— 实际执行单元

**一个工单可以拆成多个批次**，拆分方式由组长决定：

| 拆分依据 | 示例 |
|----------|------|
| 按班组拆 | 工单 2000 件 → 张三组做 1000 件（批次 1）、李四组做 1000 件（批次 2） |
| 按日期拆 | 今天做 1000 件（批次 1）、明天做 1000 件（批次 2） |
| 按产品规格拆 | 同一工单里白色做 500 件（批次 1）、黑色做 500 件（批次 2） |

**批次号规则：** `工单号-序号`，如 `WO-20260407-001-01`、`WO-20260407-001-02`

**落地操作：** 组长在 H5 上看到新工单 → 点"拆分批次" → 输入每个批次的数量和分配的班组 → 确认

### 3. 工序流转卡（Routing Card）— 跟着物料走的"身份证"

> 这是 Foxconn / BYD 等大厂电子代工厂的标准实践。他们早期也用纸质 Traveler Card，后来逐步电子化，但核心始终是 **"一卡一码 + 后台严格路由控制"**。

每个生产批次对应一张流转卡，打印成不干胶标签贴在物料托盘/周转箱上，跟着物料从第一道工序走到最后一道工序。

#### 流转卡上印什么？

**一个主二维码** + 文字信息 + 手工记录栏，三合一：

```
┌──────────────────────────────────────────────────┐
│  ⚡ 工序流转卡                                    │
│                                                   │
│  批次号：WO-20260407-001-01                       │
│  产  品：LED 模组 A                               │
│  数  量：1000 件                                  │
│  BOM 版本：V2.1                                  │
│  打印日期：2026-04-07                             │
│                                                   │
│  ┌──────────────┐                                │
│  │              │   ← 主二维码（唯一 ID）          │
│  │   QR 码      │     员工每道工序都扫这同一个码   │
│  │              │                                │
│  └──────────────┘                                │
│  批次号：WO-20260407-001-01（文字备份，断网可手输）  │
│                                                   │
│  ─────────────────────────────────────────────    │
│  序号  工序名称    单价    操作工  良品  不良  原因 │
│  ─────────────────────────────────────────────    │
│   ①   捍线     0.15元  ______  ____  ____  ____  │
│   ②   前侧     0.10元  ______  ____  ____  ____  │
│   ③   分板     0.08元  ______  ____  ____  ____  │
│   ④   贴背胶   0.12元  ______  ____  ____  ____  │
│  ─────────────────────────────────────────────    │
│  入库数量：________  入库日期：________  确认人：__ │
└──────────────────────────────────────────────────┘
```

**流转卡上的四层信息：**

| 层 | 内容 | 作用 |
|----|------|------|
| 电子层 | 主二维码（BatchID + CardSN） | 手机扫码 → 系统自动识别批次，进入报工页面 |
| 标识层 | 批次号、产品、数量、BOM 版本 | 人工识别：这是什么产品、做多少、哪个批次 |
| 路由层 | 工序列表（按顺序编号 + 单价） | 员工看一眼就知道还剩几道工序没做 |
| 备份层 | 空白表格（操作工/良品/不良/原因） | 断网、手机没电时手工记录，事后补录 |

#### 核心设计原则：一卡一码，全程复用

- 整张卡只有**一个二维码**，打印简单，员工不会扫错
- 每道工序的员工扫的是**同一个码**，不是每道工序印一个码
- 系统后台根据扫码时间 + 已完成的工序记录，自动判断当前该做哪道
- 纸质备份确保即使网络故障也不会中断生产

## 四、核心功能模块

### 模块 1：产品工序配置（管理员）

不同产品绑定不同的工序路线，灵活配置。

**示例：**

| 产品 | 工序 1 | 工序 2 | 工序 3 | 工序 4 | 工序 5 |
|------|--------|--------|--------|--------|--------|
| LED 模组 A | 捍线（0.15 元/件） | 前侧（0.10 元/件） | 分板（0.08 元/件） | 贴背胶（0.12 元/件） | — |
| LED 模组 B | 捍线（0.15 元/件） | 前侧（0.10 元/件） | 分板（0.08 元/件） | — | — |
| 电源 C | SMT 贴片（0.20 元/件） | 组装（0.25 元/件） | 测试（0.10 元/件） | 老化（0.05 元/件） | 包装（0.08 元/件） |

**功能要点：**
- 支持新增/编辑/停用产品
- 每道工序设置：工序名称、计件单价、允许损耗率（如 ≤1%）
- 工序顺序可拖拽调整
- 同一工序可分配给多个员工（多人并行生产）

**流程：**

```

组长看到待拆分的工单
    ↓

**批量拆分界面（H5）：**

拆分批次：

│                                          │
│ 已分配: 2000 / 2000  ✅                   │
│                        [确认拆分] [打印流转卡] │
└──────────────────────────────────────────┘
```

### 模块 3：扫码报工（员工 + 组长）— 后台状态机控制

#### 后台状态机逻辑（核心）

系统后台为每张流转卡维护一个**工序状态机**，严格管控工序流转：

```
流转卡 WO-001-01 的状态机：

初始状态：当前可开工序 = ①捍线

员工扫码 → 查询状态机 → 当前允许 = ①捍线 → ✅ 允许开工
组长确认 → ①标记"已完工" → 状态机推进：当前允许 = ②前侧

员工扫码 → 查询状态机 → 当前允许 = ②前侧 → ✅ 允许开工
组长确认 → ②标记"已完工" → 状态机推进：当前允许 = ③分板

如果有人试图在 ①未完成时扫卡做 ③：
员工扫码 → 查询状态机 → 当前允许 = ①捍线 → ❌ 拦截！
弹窗提示："前道工序（①捍线）尚未报工，请先完成上一道工序！"
```

**状态机的三条铁律：**

| 规则 | 说明 |
|------|------|
| 防跳序 | 上一道未完成，下一道无法开工 |
| 防重复报工 | 同一道工序已确认，不能再报第二次 |
| 防漏工序 | 必须按顺序 ①→②→③→④ 逐道完成，不能跳过任何一道 |

#### 员工操作流程（3 步，10 秒完成）

```
1. 手机扫物料上的流转卡二维码（每道工序扫同一个码）
2. 系统自动判断：
   - 前道已完成 → 显示"当前工序：②前侧 | 上道完成：995 件" → 允许开工
   - 前道未完成 → 弹窗拦截："前道工序尚未报工！" → 不允许开工
3. 做完后等待组长确认数量
```

#### 组长确认流程

```
1. H5 → "待确认"列表 → 看到本组所有待确认记录
2. 输入：良品数量 + 不良数量
3. 如果有不良品 → 选择不良原因
4. 点"确认" → 三件事同时发生：
   ✅ 报工数据入库
   ✅ 状态机推进：当前工序标记"已完工"，下一道工序自动解锁
   ✅ 实时计算：良品数 × 单价 → 即时更新员工工资
```

#### 不良原因分类（影响工资计算，必须选对）

| 原因代码 | 说明 | 影响工资 |
|----------|------|----------|
| A01 物料不良 | 来料本身有问题 | ✅ 照常计工资（不是员工的错） |
| A02 设备故障 | 机器问题导致不良 | ✅ 照常计工资 |
| A03 操作失误 | 员工自己操作不当 | ❌ 不良品不计工资 |
| A04 工艺问题 | 工艺设计不合理 | ✅ 照常计工资 |

#### 最后一道工序的特殊处理

最后一道工序组长确认后，状态机自动触发：
- 流转卡状态变为 **"待入库"**
- H5 推送通知组长/仓管员
- 仓管员扫码确认入库 → 流转卡状态变为 **"已完成"**
- 系统自动通知 ERP：该批次已完成，可入库

#### 关键规则

- 员工只能看到分配给自己班组的待做工序
- 组长确认时如果数量为 0，需填写原因（停工待料/设备故障/其他）
- **纸质 + 电子双保险**：网络不好时手工在流转卡上记录，恢复后补录系统

### 模块 4：物料损耗追踪（自动）

系统自动追踪每个批次的**投入 → 各工序产出 → 入库数量**，全链路对比。

**追踪逻辑：**

```
批次 WO-001-01 | 投料 1000 件

工序1 捍线：  投入 1000 → 良品 995  → 不良 5（物料不良）  → 损耗率 0.5% ✅
工序2 前侧：  投入  995 → 良品 990  → 不良 5（操作失误）  → 损耗率 0.5% ✅
工序3 分板：  投入  990 → 良品 940  → 不良 50（设备故障） → 损耗率 5.1% ⚠️ 超标！
工序4 贴背胶：投入  940 → 良品 938  → 不良 2（物料不良）  → 损耗率 0.2% ✅

最终入库：938 件 | 总损耗 62 件 | 总损耗率 6.2%
```

**规则：**
- 每道工序可设置**允许损耗率**（如 ≤1%），超出标红提醒
- 提醒方式：H5 消息通知 + 电脑端报表标红
- 月末自动生成损耗报表：按产品、工序、不良原因、员工维度汇总
- **重点看"操作失误"类不良** → 这是可以通过培训/管理改善的

### 模块 5：工资计算（自动）

**计算公式（大厂通例）：**

```
员工月工资 = Σ（每道工序的良品数 × 该工序计件单价）

注意：不良品不计工资。但"物料不良"和"设备故障"导致的不良除外（照常计工资）。
```

**完整公式：**

```
某道工序应得工资 = (确认良品数 + 物料不良数 + 设备故障数) × 计件单价
```

**示例：员工张三 4 月工资明细**

| 日期 | 产品 | 工序 | 良品 | 不良（物料） | 不良（操作） | 单价 | 金额 |
|------|------|------|------|-------------|-------------|------|------|
| 4/1 | 模组 A | 捍线 | 495 | 5 | 0 | 0.15 | 75.00 |
| 4/1 | 模组 A | 捍线 | 475 | 0 | 5 | 0.15 | 71.25 |
| 4/2 | 模组 B | 捍线 | 518 | 2 | 0 | 0.15 | 78.00 |
| 4/2 | 模组 A | 前侧 | 298 | 2 | 0 | 0.10 | 30.00 |
| ... | ... | ... | ... | ... | ... | ... | ... |
| **合计** | | | | | | | **4,820.00** |

> 说明：4/1 下午那行，5 件不良是"操作失误"，所以这 5 件不计工资（475 × 0.15 = 71.25，而不是 480 × 0.15 = 72.00）

**功能要点：**
- 支持按月、按周、按日查看工资汇总
- 员工端只能看自己的，组长能看本组的，管理员能看全部
- 支持导出 Excel（发给财务）
- 月末一键同步到 ERP

### 模块 6：ERP 数据对接

| 数据方向 | 内容 | 频率 |
|----------|------|------|
| ERP → 工序系统 | 产品基础信息、BOM 物料清单 | 实时 API |
| 工序系统 → ERP | 触发领料需求（工单确认后自动生成） | 实时 |
| 工序系统 → ERP | 工序完工数据、入库数量 | 每日同步 |

**核心 API 接口：**

| 接口 | 方向 | 说明 |
|------|------|------|

| `POST /api/erp/wage-settlement` | → ERP | 推送月度工资汇总 |
| `GET /api/erp/products` | ← ERP | 拉取产品信息和 BOM |

## 六、流转卡打印与扫码设计

### 二维码内容

流转卡只有一个主二维码，内容是唯一 ID 的访问链接：

```
https://your-domain.com/b/WO-20260407-001-01
```

- 员工扫码 → 系统查状态机 → 自动判断当前允许哪道工序 → 进入对应页面
- 二维码下方同时印批次号文字，断网时可在 H5 手动输入
- 整个流程只有这一个码，不会混淆

### 流转卡打印规范

- 系统自动生成 PDF，普通打印机 + 不干胶标签纸即可
- 建议尺寸：**10cm × 15cm**，一张 A4 排 4 张
- 纸张选择：铜版纸不干胶（防水、防撕、粘得牢）
- 打印内容：主二维码 + 批次号（文字）+ 产品信息 + BOM 版本 + 工序列表 + 空白记录栏

### 断网 / 异常处理

| 场景 | 处理方式 |
|------|----------|
| 网络正常 | 扫码 → 系统自动走状态机 → 报工入库全线上 |
| 网络中断 | 手工在流转卡空白栏记录（操作工/数量/时间/不良原因） |
| 恢复网络 | 组长在 H5 上补录数据，勾选"补录"标记 |
| 手机没电 | 同上，用纸质记录，事后补录 |
| 流转卡丢失 | 组长在系统里输入批次号 → 重新打印（系统记录原始卡号，追溯不断） |

## 七、技术建议

| 项目 | 建议 |
|------|------|
| 前端（H5） | Vue 3 / React，移动端优先 |
| 后端 | 跟你们 ERP 统一技术栈 |
| 数据库 | MySQL / PostgreSQL |
| 二维码生成 | qrcode.js，前端直接生成并打印 |
| 流转卡打印 | jsPDF 生成 PDF，普通打印机即可 |
| 部署 | 和 ERP 同服务器，内网访问 |

## 八、落地步骤（4 个阶段）

| 阶段 | 内容 | 时间 |
|------|------|------|
| **第一阶段：流程跑通** | 配置 1-2 个产品的工序路线 + 录入员工 + 电脑端手动报工（不走扫码） | 1 周 |
| **第二阶段：流转卡上线** | 工单拆批次 + 打印流转卡 + 手机扫码报工 + 组长确认 | 1-2 周 |
| **第三阶段：数据闭环** | 物料损耗追踪 + 不良原因分类 + 工资自动计算 | 1 周 |
| **第四阶段：对接 ERP** | API 开发 + 领料单自动生成 + 入库回写 + 工资同步 | 1-2 周 |

> 💡 **从第一阶段开始**：先在电脑上手动录入跑通整个流程，确认工序配置、报工逻辑、工资计算都没问题，再加扫码和 ERP 对接。小步快跑，逐步完善。
