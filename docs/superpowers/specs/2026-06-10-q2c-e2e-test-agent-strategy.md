# agent-browser 多角色自动化测试策略

> **所属方案**：生产型 ERP 全链路 E2E 测试
> **本文档定义**：Agent 角色定义、接力测试机制、异常与逆向操作模拟

---

## 1. Agent 角色定义

### 1.1 角色配置表

每个 Agent 对应一个系统角色，携带独立的浏览器会话（Cookie / Session）和角色权限。

| Agent ID | 角色 | 系统账号 | 权限范围 | 参与阶段 |
|----------|------|----------|----------|----------|
| Agent-S1 | 销售专员 | `sales_user` | 报价 CRUD、订单 CRUD、发货申请 | Phase 1, 4 |
| Agent-S2 | 销售经理 | `sales_manager` | 报价审批、发货审批、信用审批 | Phase 1, 4 |
| Agent-P1 | 计划员 | `planner` | BOM 查看、MRP 运算、需求分解 | Phase 2 |
| Agent-PU1 | 采购专员 | `buyer` | 采购申请、询价、PO 管理、到货登记 | Phase 3A |
| Agent-PU2 | 采购经理 | `purchase_manager` | 采购审批 | Phase 3A |
| Agent-M1 | 生产主管 | `production_manager` | 工单下达、报废审批 | Phase 3B |
| Agent-M2 | 车间操作员 | `shop_operator` | 领料确认、报工录入 | Phase 3B |
| Agent-Q1 | 质检员 | `qc_inspector` | 来料检验、成品质检、MRB 参与 | Phase 3A/3B |
| Agent-W1 | 仓管员 | `warehouse_keeper` | 入库、出库、拣货、盘点 | Phase 3A/3B/4 |
| Agent-F1 | 财务会计 | `accountant` | 应收/应付、开票、核销 | Phase 5 |
| Agent-F2 | 成本会计 | `cost_accountant` | 成本核算、成本分析 | Phase 5 |
| Agent-F3 | 出纳 | `cashier` | 收款、付款 | Phase 5 |
| Agent-F4 | 总账会计 | `gl_accountant` | 总账结算、报表、期间结账 | Phase 5 |
| Agent-QM1 | 质量主管 | `qc_manager` | MRB 评审、报废审批（质量侧） | Phase 3A/3B |
| Agent-GM | 总经理 | `gm` | 高金额审批终审 | 跨域 |

> **说明**：`gl_accountant` 和 `qc_manager` 如系统中尚未创建，需在测试用户初始化脚本中补充。总账会计可复用 `accountant` 账号增加总账权限，质量主管可复用 `qc_inspector` 账号增加 MRB 权限——取决于实际系统配置。

### 1.2 Agent 能力定义

每个 Agent 需要具备以下基础能力：

```
Agent 能力 = {
  1. 页面导航：能访问其权限范围内的所有页面
  2. 表单填写：能填写表单字段（文本、下拉、日期、数量）
  3. 按钮操作：能点击"提交"、"审批通过"、"驳回"等操作按钮
  4. 数据读取：能从页面表格、详情页读取关键数据（单号、金额、状态）
  5. 状态断言：能验证页面上的状态标签、金额、数量是否符合预期
  6. 等待与轮询：能等待页面加载完成、等待异步任务（如 MRP 运算）完成
  7. 错误处理：能识别并报告页面错误（弹窗、错误提示、403/500）
}
```

### 1.3 Agent 初始化流程

```
对每个 Agent:
  1. 打开浏览器（独立 Session）
  2. 访问登录页面
  3. 输入用户名/密码 → 登录
  4. 验证登录成功（检查仪表盘是否加载）
  5. 记录 Session Token（Cookie）
  → Agent 就绪
```

## 2. 接力测试机制（核心设计）

### 2.1 接力原理

Quote-to-Cash 是多角色顺序操作的业务流。Agent A 完成节点 1 后，Agent B 需要在系统中找到 A 创建的数据，继续操作节点 2。这就是**接力测试**。

```
接力模型:
  Agent-S1 创建 SO → 输出: SO 单号 = "SO-2026-001"
       ↓ 接力传递: SO 单号
  Agent-P1 搜索 SO 单号 → 执行 MRP → 输出: 采购需求 + 工单建议
       ↓ 接力传递: 采购需求 ID + 工单号
  Agent-PU1 查看采购需求 → 创建 PR → ...
       ↓
  ...（持续接力直到 Phase 5 结束）
```

> **数据创建方式说明**：本方案中，**业务数据**（报价/SO/PO/工单等）全部通过 agent-browser 的 UI 操作创建——这是 E2E 测试的核心价值：模拟真实用户行为。**主数据**（物料/BOM/客户/供应商/仓库/用户）则通过 SQL 脚本预置，不经过 UI——因为主数据创建不是 Q2C 流程的一部分，且数量大、UI 操作繁琐。详见《测试数据准备与环境隔离》文档。

### 2.2 状态传递协议

#### 传递数据结构

```json
{
  "run_id": "q2c-20260610-run001",
  "phase": "phase_1_sales",
  "current_node": "S5",
  "status": "completed",
  "artifacts": {
    "quotation_id": "QT-2026-001",
    "contract_id": "CT-2026-001",
    "sales_order_id": "SO-2026-001",
    "customer_id": "CUS-001",
    "product_ids": ["PRD-FG-001"],
    "total_amount": 150000.00
  },
  "context": {
    "customer_name": "测试客户A",
    "product_name": "成品A",
    "quantity": 100,
    "unit_price": 1500.00,
    "discount_rate": 0.10,
    "tax_rate": 0.13
  },
  "next_agent": "Agent-P1",
  "next_action": "execute_mrp"
}
```

#### 接力检查点

每个节点完成后，接力数据必须通过以下检查：

| 检查项 | 说明 |
|--------|------|
| **单号存在** | 生成的单号在系统中可查到 |
| **状态正确** | 单据状态符合预期（如 SO 应为"已确认"） |
| **数据完整** | 明细行数据与输入一致（产品、数量、金额） |
| **下游可见** | 下一个 Agent 能在其工作台中看到待办 |
| **事件已发** | 对应的 Event Bus 事件已成功投递 |

### 2.3 接力执行模式

#### 模式 A：串行接力（Happy Path）

```
Agent-S1 → Agent-S2 → Agent-S1 → Agent-P1 → Agent-PU1 → Agent-PU2
→ Agent-PU1 → Agent-W1 → Agent-Q1 → Agent-W1 → Agent-M1 → Agent-W1
→ Agent-M2 → Agent-Q1 → Agent-W1 → Agent-W1 → Agent-W1 → Agent-F1
→ Agent-F2 → Agent-F1 → Agent-F3 → Agent-F1
```

**执行规则：**
1. 每个 Agent 完成操作后，将输出数据写入共享接力文件
2. 下一个 Agent 启动后，先读取接力文件获取上游数据
3. 在系统中通过搜索/筛选定位到具体单据
4. 执行操作并断言结果
5. 更新接力文件，传递给下一个 Agent

#### 模式 B：并行分支（采购 + 生产同时进行）

```
                    Agent-P1（MRP 完成）
                    ┌─────────┴──────────┐
            Agent-PU1（采购流程）    Agent-M1（生产流程）
            Agent-PU2（采购审批）    Agent-W1（领料）
            Agent-PU1（PO 发送）     Agent-M2（报工）
            Agent-W1（收货入库）     Agent-Q1（质检）
            Agent-Q1（来料检验）     Agent-W1（成品入库）
                    └─────────┬──────────┘
                    Agent-W1（发货出库）
```

**执行规则：**
1. MRP 完成后，采购分支和生产分支的 Agent 同时启动
2. 两条分支各自独立执行，各自维护接力文件（`relay-purchase.json` 和 `relay-production.json`）
3. 两条分支都完成后，汇聚到发货阶段
4. **汇聚同步机制**：汇聚点前执行屏障检查——验证两个分支的接力文件都标记为 `status: completed`。若任一分支未完成，等待最长 5 分钟，超时则标记测试失败
5. **冲突避免**：两条分支共享同一个仓管员 Agent（Agent-W1）。当两条分支都需要仓管操作时，采用串行排队——采购分支优先（先到先服务），生产分支排队等待
4. 汇聚前需验证：采购物料已入库 + 生产成品已入库

#### 模式 C：异常注入（在 Happy Path 基础上注入异常）

```
正常流程中，在指定节点注入异常条件：
  - 替换数据（如：将合格品数量改为 0，触发质检不合格）
  - 修改参数（如：将折扣率设为 20%，触发审批）
  - 阻断操作（如：审批时选择"驳回"而非"通过"）
```

### 2.4 Agent 间协作示例

#### 示例 1：销售订单 → MRP 运算

```
[Agent-S1 操作]
  1. 登录 → 进入销售订单列表
  2. 点击"新建销售订单"
  3. 选择客户（从接力文件获取 customer_id）
  4. 添加明细行：产品 PRD-FG-001，数量 100
  5. 填写交付日期
  6. 点击"提交"
  7. 断言：页面显示"创建成功"，获取 SO 单号
  8. 将 SO 单号写入接力文件

[接力文件更新]
  artifacts.sales_order_id = "SO-2026-001"
  next_agent = "Agent-P1"
  next_action = "execute_mrp"

[Agent-P1 操作]
  1. 登录 → 进入 MRP 运算页面
  2. 搜索 SO 单号 "SO-2026-001"
  3. 确认 SO 明细（产品、数量、交期）
  4. 点击"执行 MRP 运算"
  5. 等待运算完成（轮询状态）
  6. 查看运算结果：采购需求 N 条，生产工单建议 M 条
  7. 将需求 ID 写入接力文件
```

#### 示例 2：审批接力

```
[Agent-S1 提交审批]
  1. 创建报价单，折扣率设为 20%（超过 15% 阈值）
  2. 提交 → 断言：页面显示"已提交审批"
  3. 写入接力文件：quotation_id, approval_pending = true

[Agent-S2 执行审批]
  1. 登录 → 进入审批待办列表
  2. 搜索报价单号 "QT-2026-001"
  3. 点击查看详情
  4. 断言：折扣率 = 20%，金额明细正确
  5. 点击"审批通过"
  6. 断言：报价状态变为"已生效"
  7. 更新接力文件：approval_status = "approved"
```

## 3. 异常与逆向操作模拟

### 3.1 异常场景分类

| 类别 | 说明 | 示例 |
|------|------|------|
| **业务异常** | 正常业务流程中的非预期情况 | 库存不足、信用超限、供应商延迟 |
| **审批异常** | 审批流中的非预期操作 | 驳回、超时、撤回、委托 |
| **质量异常** | 质检不合格的处理 | 来料不合格、成品报废、让步接收 |
| **逆向操作** | 业务回退操作 | 退货、退款、红字冲销、工单取消 |
| **数据异常** | 数据层面的异常 | 金额不匹配、数量超出范围 |

### 3.2 异常注入策略

#### 策略 1：数据驱动注入

在测试数据中预置异常条件：

```
异常测试数据集:
  - discount_above_threshold: { rate: 0.20 }  # 触发报价审批
  - credit_limit_exceeded: { amount: 999999 }  # 触发信用冻结
  - insufficient_stock: { qty: 99999 }         # 触发缺料
  - quality_reject: { defect_rate: 0.5 }       # 触发质检不合格
```

#### 策略 2：Agent 行为注入

在 Agent 操作步骤中注入异常行为：

```
正常步骤: click("审批通过")
注入异常: click("驳回") + fill("驳回原因", "价格偏高，请重新协商")
```

#### 策略 3：环境注入

修改系统配置来触发异常：

```
- 将安全库存参数设为极高 → MRP 触发大量补货建议
- 将审批超时参数设为 1 分钟 → 快速触发超时升级
```

### 3.3 逆向操作场景

| 逆向操作 | 触发节点 | 操作流程 | 验证要点 |
|----------|----------|----------|----------|
| **报价撤回** | S2 → S3 | 提交人撤回待审批报价 | 审批待办清除；报价恢复为"草稿" |
| **订单取消** | S5 | 已确认的 SO 执行取消 | 库存预留释放；MRP 需求取消；下游工单/PO 取消 |
| **采购退货** | PU6 | 收货后退货给供应商 | 库存扣减；AP 冲减；供应商退款 |
| **工单取消** | M1 | 已下达工单取消 | 退料入库；释放产能；通知计划员 |
| **生产报废** | M4 | 质检不合格判定报废 | 报废品入废品仓；成本重算；工单剩余数量调整 |
| **销售退货** | W4 → W3 | 客户签收后退货 | 库存回仓（待检）；AR 冲减；退款或换货 |
| **红字冲销** | F3 | 开票后冲销（退货/折让） | 红字发票；AR 负数冲抵；税务调整 |
| **付款冲销** | FP3 | 付款后发现问题 | 银行退回；AP 恢复；重新发起付款 |

### 3.4 异常场景的 Agent 脚本结构

```yaml
# 示例：质检不合格 → MRB 评审 → 让步接收
scenario: quality_rejection_with_concession
phase: 3B
inject_point: M4  # 在生产质检节点注入

steps:
  - agent: Agent-Q1
    action: 执行成品质检
    inject: 检验结果 = 不合格，缺陷率 30%
    assert:
      - 质检报告标记为"不合格"
      - 系统自动触发 MRB 评审
      - 不合格品移入隔离仓

  - agent: Agent-M1
    action: 查看 MRB 待办
    assert:
      - MRB 列表中有新评审项
      - 关联工单号正确
      - 缺陷描述完整

  - agent: Agent-GM
    action: MRB 评审决定 → 让步接收
    fill:
      - 处置决定: 让步接收
      - 客户通知: 已通知
      - 价格调整: 折扣 5%
    assert:
      - MRB 状态 → 已关闭
      - 隔离仓库存转回成品仓
      - 工单成本重算（扣除折扣）
      - 通知销售经理（价格调整）
```

### 4.3 断言机制

| 断言类型 | 说明 | 示例 |
|----------|------|------|
| **页面断言** | 验证页面元素状态 | 状态标签显示"已审批"；金额字段 = 150000 |
| **数据断言** | 验证数据库数据 | SO 状态字段 = confirmed；库存数量 = 100 |
| **事件断言** | 验证事件是否触发 | Event Bus 中存在 order_created 事件 |
| **通知断言** | 验证通知是否发送 | 计划员收到站内信，内容含 SO 单号 |
| **财务断言** | 验证会计分录平衡 | AR 借方 = 发票金额；AR 贷方 = 收款金额 |
| **审计断言** | 验证审计日志 | 操作记录中包含"审批通过"动作 |
