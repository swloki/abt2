## 2. 测试数据注入策略

### 2.1 注入方式

| 方式 | 适用场景 | 优缺点 |
|------|----------|--------|
| **数据库直插** | 主数据初始化（物料、BOM、客户、供应商） | ✅ 快速；❌ 跳过业务校验 |
| **API 调用** | 业务数据创建（通过 abt-web 的 HTTP 接口） | ✅ 完整业务校验；✅ 贴近真实操作 |
| **UI 操作（agent-browser）** | 测试场景本身需要操作的节点 | ✅ 真实模拟用户行为；❌ 速度慢 |
| **SQL 脚本** | 环境重置、数据清理 | ✅ 快速重置；❌ 仅用于清理 |

**推荐策略**：
- **主数据**（物料/BOM/客户/供应商/仓库/用户）→ 数据库直插（通过 Migration 脚本）
- **业务数据**（报价/SO/PO/工单）→ agent-browser UI 操作（作为测试场景的一部分）
- **环境重置** → SQL 脚本清理 + 主数据重新注入

### 2.2 数据注入脚本结构

```
tests/e2e-q2c/fixtures/
├── 01_master_data.sql        # 主数据：物料、BOM、客户、供应商、仓库、价格
├── 02_users_and_roles.sql    # 用户、角色、权限
├── 03_initial_inventory.sql  # 初始库存
├── 04_system_config.sql      # 系统参数（审批阈值、安全库存等）
└── 99_cleanup.sql            # 清理脚本（回滚所有测试数据）
```

### 2.3 测试数据标识

所有测试数据统一使用标识前缀，便于区分和清理：

| 数据类型 | 前缀 | 示例 |
|----------|------|------|

| 供应商 | `TEST-` | `TEST-供应商A` |

## 3. 环境隔离策略

### 3.1 隔离层级

| 隔离方式 | 描述 | 适用场景 | 成本 |
|----------|------|----------|------|
| **数据库级隔离** | 独立 PostgreSQL Schema（`test_q2c`） | 默认策略 | 低 |
| **数据级隔离** | 同一数据库，通过 `operator_id` 和数据前缀区分 | 资源有限时 | 最低 |
| **实例级隔离** | 独立的 Docker Compose 环境 | CI/CD 环境 | 中 |

### 3.2 推荐方案：数据库 Schema 隔离

```sql
-- 创建测试 Schema
CREATE SCHEMA test_q2c;

-- 在测试 Schema 中创建相同的表结构
-- (通过 Migration 脚本自动完成)

-- 测试连接串
DATABASE_URL=postgres://abt:test@localhost:5432/abt_v2_test?currentSchema=test_q2c
```

**优点**：
- 同一 PostgreSQL 实例，资源消耗小
- 完全隔离，测试数据不影响开发数据
- 清理简单：`DROP SCHEMA test_q2c CASCADE`

### 3.3 测试执行前后的环境管理

#### 测试前（Setup）

```
1. 创建测试 Schema（如不存在）
2. 执行 Migration 创建表结构
3. 注入主数据（01-04 SQL 脚本）
4. 启动 abt-web（指向测试 Schema）
5. 初始化所有 Agent（登录）
6. 验证环境就绪（健康检查）
```

#### 测试后（Teardown）

```
1. 生成测试报告
2. 截图存档
3. 数据快照（可选，用于问题排查）
4. 清理测试数据（99_cleanup.sql 或 DROP SCHEMA）
```

### 3.4 并发测试支持

如果多个测试场景需要并行运行：

| 策略 | 说明 |
|------|------|
| **Schema 隔离** | 每个并行测试使用独立 Schema（`test_q2c_run1`, `test_q2c_run2`） |
| **数据隔离** | 同一 Schema，通过 `run_id` 标记数据，清理时按 `run_id` 删除 |
| **端口隔离** | 每个 abt-web 实例使用不同端口（8001, 8002, ...） |

## 4. 数据快照与回滚

### 4.1 关键节点快照

在以下节点创建数据快照，便于异常时回溯：

| 快照点 | 时机 | 内容 |
|--------|------|------|
| SNAP-0 | 测试开始前 | 主数据基线 |
| SNAP-1 | SO 创建后 | 订单数据 + 库存预留状态 |
| SNAP-2 | MRP 完成后 | 需求分解结果 + 库存快照 |
| SNAP-3 | 生产完工后 | 工单成本 + 库存快照 |
| SNAP-4 | 发货完成后 | 库存快照 + AR 凭证 |
| SNAP-5 | 结算完成后 | 全量数据快照（用于最终对账） |

### 4.2 快照实现

```sql
-- 关键表的数据快照（插入到 _snapshot 后缀的镜像表）
INSERT INTO inventory_snapshot (run_id, snapshot_point, warehouse, product, qty, status, created_at)
SELECT 'run001', 'SNAP-3', warehouse, product, qty, status, NOW()
FROM inventory;
```

## 5. 测试数据管理最佳实践

1. **幂等性**：主数据注入脚本可重复执行（使用 `ON CONFLICT DO NOTHING` 或 `UPSERT`）
2. **可追溯性**：所有测试数据通过 `operator_id` 关联到测试用户
3. **可清理性**：所有测试数据可通过统一脚本一键清理
4. **真实性**：测试数据使用真实业务场景的数量级和比例
5. **独立性**：每个测试场景的数据自包含，不依赖其他场景的残留数据
