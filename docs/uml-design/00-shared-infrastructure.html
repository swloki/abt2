功能

为所有业务单据和产品生成唯一编号。支持两种策略：Sequential（默认，{前缀}-{年}-{月}-{序号}，如 SO-2026-05-00142）和 Timestamp（x+Unix时间戳，如 x1747891200，用于产品编码）。策略由 DocumentSequence.strategy 字段按 prefix 配置

解决什么问题

杜绝各模块自行编号导致的编号冲突、格式混乱、无法追溯。按日期分段确保每月序号从 001 重新开始

使用场景（按模块）

Sales: QUO/SO/SR/SRT/REC
Purchase: PQ/PO/PRT/PREC/PAY/MISC
WMS: AN/MR/BF/CC/TRF/FC/LCK
MES: PP/WO/WR/OO/PI/PR
OM: OO/OT
QMS: QS/QR/MRB/RMA
FMS: CJ/WO/ER
MasterData: PRODUCT (Timestamp策略)

  source_id: i64

  target_id: i64

enum LinkType {
  DERIVED_FROM // 报价→订单
  TRIGGERS // 订单→发货申请
  REFERENCES // 退货→原发货
  RECONCILES // 对账单→发货
  INSPECTS // 检验→来料通知
  FULFILLS // 采购入库→采购单
  ALLOCATES // 领料→工单
  }

功能

以有向图的方式记录任意两个业务单据之间的关系。支持图遍历查询，可追溯一张报价单到最终发票的完整链路

解决什么问题

模块间单据关系零散记录在各表里，事后对齐几乎不可能。统一图谱后：一键追溯、BI 报表、ISO 9001 合规审计

批量接口

create_links — 批量创建关联（单条传 vec![one]），订单确认时一次性创建多个。返回 BatchResult。查询接口 find_linked 返回 PaginatedResult，支持分页

路径加速（path + depth）

path 采用物化路径模式（如 QUO.1.SO.42.SR.15），depth 记录链路层级。创建关联时自动计算：path = parent_path + "." + current。单次 LIKE 查询即可获取整条链路，无需递归。适用于质量追溯、批次追溯等深层链路场景

使用场景

Sales: 报价→订单→发货→对账→应收
Purchase: 采购单→来料→IQC→入库→对账→付款
MES: 计划→工单→报工→报检→入库
OM: 委外单↔工单 | 委外单↔追踪节点 | 委外单↔调拨单
QMS: RMA→工单→工序→来料批次（全链路质量追溯）
FMS: 日记账↔核销↔出库单/入库单

③ 库存预留层 (InventoryReservation)

entity InventoryReservation {
  id: i64 PK
  product_id: i64 FK→products
  warehouse_id: i64 FK→warehouses
  reserved_qty: Decimal(18,6)
  reservation_type: ReservationType
  source_type: DocumentType
  source_id: i64
  source_line_id: i64? // 单据行级精确释放
  status: ReservationStatus
  priority: i32 // 预留优先级
  expires_at: Timestamp? // TTL自动过期
  created_at: Timestamp
}

enum ReservationType {
  HARD // 工单锁定，不可抢占
  SOFT // 销售预留，可被高优先级抢占
  SAFETY_STOCK // 安全库存永久预留
  }
enum ReservationStatus {
  ACTIVE, FULFILLED, CANCELLED, EXPIRED
  }

功能

区分「现有量」和「可用量」。提供预留、释放、过期自动清理机制。
可用量 = 现有量 − 已预留量

解决什么问题

当前系统只有库存总量，没有可用量（ATP）。导致：销售接单不知道能不能发、采购不知道该买多少、工单领料时才发现库存不够

批量接口

reserve — 批量预留（单条传 vec![one]），单次事务内完成，返回 BatchResult。total_reserved 支持按 warehouse_id 过滤，精确计算仓库级可用量

并发控制

reserve() 使用 SELECT ... FOR UPDATE 锁定相关库存行，防止同一产品并发预留导致超卖。total_reserved() 在事务内加锁计算，保证可用量读取的强一致性

使用场景

Sales: 订单确认→SOFT预留（TTL 7天）
MES: 工单下达→HARD预留（直到领料或完工）
Purchase: 到货→释放安全库存预留
WMS: 领料→消耗HARD | 盘点→校准预留
采购建议 = 需求量 − 现有量 − 已预留

功能

统一成本分类账。每个产生成本的事件写入同一张表，遵循双层记账原则（每个操作同时记借方和贷方）

解决什么问题

传统 ERP 陷阱：运营模块先建、财务后加，事后对账痛苦。从第一天起就记录成本，FMS 报表、利润中心 P&L、毛利分析全部从这一张表查询

功能

跨模块解耦的核心机制。Outbox 模式：业务事务内将事件写入 domain_events 表（Outbox），后台 EventProcessor 通过 LISTEN/NOTIFY 实时消费并异步分发给注册的 Handler

接口改进

publish 改为接受 EventPublishRequest：参数封装为 struct，idempotency_key 为 Option，传 None 自动生成 {aggregate_type}:{aggregate_id}:{event_type}，也可自定义覆盖。底层 INSERT ON CONFLICT DO NOTHING 防重复发布
mark_processed 统一批量：单条传 [id]，批量传 [id1, id2, ...]，减少 DB 交互
find_events 统一查询：支持 aggregate_type、event_type、since 等可选过滤，返回 PaginatedResult
EventProcessor 生命周期：start() / stop() 替代 run()→JoinHandle，内部自行 spawn；is_running() + last_processed_at() 提供健康检查
process_event 内部化：不再暴露在公开接口中，仅 EventProcessor 内部调用

解决什么问题

模块间直接调用导致的紧耦合：新增下游逻辑必须改上游 Service。事件驱动后，新逻辑只需注册 handler，上游代码零改动。Outbox 保证：事件与业务数据在同一事务内写入，零丢失风险

Sales: SalesOrderConfirmed / Shipped / ReturnReceived
Purchase: POConfirmed / ArrivalReceived / PaymentPaid
MES: PlanReleased / WOReleased / WOCompleted / ReceiptConfirmed

QMS: InspectionPassed / Failed / MRBDispositioned / RMACreated
FMS: CashJournalConfirmed / WriteOffCompleted

Handler 实现示例（注册到 Registry）

同步强一致：库存预留/释放、质量关卡硬门 — 调用者事务内直接执行
异步 Outbox：成本记录、文档链接、Workflow 触发、通知 — 后台异步消费

InventoryReservationHandler → 预留/释放库存 [同步]
CostEntryHandler → 记录成本分录 [异步]

WorkflowTriggerHandler → 触发工作流 [异步]
StateTransitionHandler → 触发状态转换 [异步]

Outbox 工作流（异步 Handler）

写入: 业务事务内 INSERT domain_events (status=PENDING) + NOTIFY
消费: EventProcessor 收到 LISTEN → FETCH FOR UPDATE SKIP LOCKED
轮询兜底: 每 30s 扫描 status=PENDING 且 created_at < now()-30s 的事件，防止 NOTIFY 丢失（NOTIFY 快速通道 + 轮询双保险）
分发: EventHandlerRegistry.dispatch() → 异步 Handler 执行
完成: 成功 → mark_processed() | 失败 → mark_failed() + 指数退避重试
死信: retry_count > max_retries → DEAD_LETTER（通过 DeadLetterService 人工干预：list_dead_letters 查询、retry_one 重试、archive 归档）

关键约束：[TransactionMode: 混合模式] Outbox 模式 — 业务事务内写入事件 | EventProcessor 后台异步消费（start/stop 生命周期管理，process_event 内部化）| LISTEN/NOTIFY 低延迟驱动 + 定期轮询兜底（每 30s 扫描超时事件，防 NOTIFY 丢失） | 失败重试（指数退避）+ 死信队列（DeadLetterService: list_dead_letters / retry_one / archive） | publish idempotency_key 改为 Option（None 自动生成，可覆盖）防重复发布 + IdempotencyService 防重复消费 | mark_processed 统一批量版 | find_events 统一查询（支持多维度过滤） | Handler 独立事务，失败不影响发布方 | 查询统一返回 PaginatedResult

功能

统一管理所有业务单据的生命周期状态。状态定义和转换规则存 DB 可动态配置，guard 条件复用 Condition AST。side_effects 使用 SideEffect 枚举（PublishEvent / Notify / TriggerWorkflow / UpdateField），类型安全，存储为 JSONB

解决什么问题

transition() 校验流程

1. 查询当前状态（from_state）
2. 匹配转换规则（from_state → to_state）
3. 校验 guard_condition（如果存在）
4. 插入 EntityStateLog（追加写）
5. 执行 side_effects（发布事件等）

PurchaseOrder: Draft → Confirmed → PartiallyReceived → Received → Closed

设计原则：定义存 DB 可动态配置 | guard 复用 Condition AST | EntityStateLog 追加写不可变 | 替换分散的 if-else 状态校验 | get_allowed_transitions 供 UI 运行时查询（替代 can_transition） | get_state_history 支持分页

⑧ 幂等去重服务 (IdempotencyService)

entity IdempotencyRecord {
  id: i64 PK BIGSERIAL
  idempotency_key: String UNIQUE "{event_id}:{handler}"
  event_id: i64 FK→domain_events
  handler_name: String
  status: String // Processing / Processed
  result: JSONB? // handler 输出（可选）
  created_at: Timestamp
  expires_at: Timestamp? // 过期清理
}

trait IdempotencyService {
  check_and_mark(event_id, handler): bool
  mark_processed(event_id, handler, result?)
  cleanup_expired(before): u64
}

功能

为系统提供 三级幂等保障：API 级（防重复提交）、Handler 级（Outbox 重试幂等）、补偿级（Workflow Saga 重试幂等）。即使同一操作被多次触发，也只执行一次

解决什么问题

Outbox 重试：EventProcessor 崩溃后事件被重新投递 → 防成本记双份、文档链接重复
API 重复提交：前端双击/网络重试 → 防重复创建订单/发货单
Workflow 补偿：Saga 重试时 → 防补偿操作重复执行

工作流

1. EventProcessor 取出事件 → 准备分发给 Handler
2. check_and_mark(event_id, handler_name)
   → true: 首次执行，继续
   → false: 已处理过，跳过
3. Handler 执行
4. mark_processed() 记录结果
5. 定期 cleanup_expired() 清理过期记录

设计原则：三级幂等（API + Handler + Saga补偿）| idempotency_key UNIQUE 原子性 | 与 domain_events.idempotency_key 互补（事件级 vs 操作级）| 过期记录定期清理
两层幂等 key 说明： DomainEvent.idempotency_key = 事件级去重（publish 传 None 时自动生成 {aggregate_type}:{aggregate_id}:{event_type}，也可调用方自定义覆盖，防同一事件重复写入 Outbox）| IdempotencyRecord.idempotency_key = "{event_id}:{handler}"（处理级去重，防同一事件被同一 handler 重复消费）

功能

ServiceContext：服务调用上下文，包装 PgExecutor + operator_id + department_id + data_scope + trace_id + request_id。data_scope (All/Department/Self) 由 Auth 服务从 JWT 解析注入，支持行级数据权限（RLS）。所有共享服务接受 ServiceContext 作为第一参数，消除逐参数传递 operator_id / trace_id 的冗余。共享服务（如 AuditLog.record、EventBus.publish）在调用者事务内执行，保证强一致

DomainError：基于 thiserror 定义结构化错误枚举，自动派生 Display + Error trait。新增 Validation 变体用于输入校验，Internal 的完整错误链仅记录到日志（含 Error::source 链追溯根因），不泄露给客户端。ServiceError 标记 #[deprecated]，新代码统一使用 DomainError

PaginatedResult：所有查询接口的标准分页返回结构。PageParams：所有分页查询的标准输入参数（page + page_size）

BatchResult：所有 batch_* 方法的统一返回结构。mode 标识语义：Atomic（全成功或全失败，如 batch_create_link、batch_create 成本分录）或 ContinueOnError（部分失败继续，如 batch_reserve）。success_count 记录成功数，failed_items 记录每个失败项的位置和原因

PgExecutor 设计要点

ServiceContext 封装 — PgExecutor 不再直接传入 service，而是包装在 ServiceContext 中，携带 operator_id / trace_id / request_id
TransactionMode 声明 — 每个共享服务方法明确声明事务模式（InCallerTx / IndependentTx / AsyncOutbox），编译器辅助检查
强一致保证 — InCallerTx 模式下 AuditLog.record + EventBus.publish + StateMachine.transition 全部在同一事务内完成

迁移策略

ServiceContext：在 abt/src/repositories/mod.rs 定义，所有共享服务接受 ServiceContext 作为第一参数
新服务：共享基础设施组件返回 Result<T, DomainError>，接受 ServiceContext
老服务：ServiceError 标记 deprecated，anyhow::Result 保持兼容

设计原则：ServiceContext 封装 PgExecutor + 操作元数据（operator_id / trace_id / request_id）| TransactionMode 显式声明事务模式（InCallerTx / IndependentTx / AsyncOutbox）| DomainError 统一错误模型（ServiceError deprecated）| thiserror 自动 Display + Error | #[from] anyhow::Error 兼容老代码 | PaginatedResult 查询统一返回 | BatchResult batch_* 统一返回（Atomic 全建全不建 | ContinueOnError 部分失败继续）

请求鉴权（Interceptor 统一拦截）:
gRPC interceptor 从 metadata 提取 token
→ 验证 JWT 签名 + 过期时间 → 构建 AuthContext → 注入 request extensions
→ 所有请求必须经过 interceptor，禁止绕过

权限校验（路由权限表自动拦截）:
PERMISSION_ROUTE_MAP: HashMap<method_path, (resource, action)>
  如 "/abt.v1.ProductService/UpdateProduct" → ("PRODUCT", "WRITE")
  如 "/abt.v1.SalesOrderService/ListSalesOrders" → ("SALES_ORDER", "READ")
interceptor 自动查表 → PermissionService.check_permission → 放行或 PermissionDenied
新增接口只需在 PERMISSION_ROUTE_MAP 加一行，handler 无需关心权限逻辑

PermissionService.check_permission(user_id, resource, action)
→ RolePermissionCache.get_merged_permissions(role_ids) 内存查询（不查 DB）
→ super_admin 直接放行 → 校验失败返回 PermissionDenied，不进入业务逻辑
缓存启动时全量加载 + 角色变更时自动 reload，无 Redis 依赖

设计要点

RBAC 模型: User → Role → Permission (resource:action)
用户可拥有多角色，角色可拥有多权限

资源定义: 代码注册（collect_all_resources），非数据库表
权限格式: "PRODUCT:READ", "BOM:WRITE"
单体 Rust 应用，新资源随新功能一起部署，代码注册足够

超级管理员: is_super_admin 或 system_role="super_admin"
绕过所有权限检查

JWT 原则：只存 ID，不存详情
Claims 存 sub/role_ids/department_ids 等轻量标识符（当前 ~300 字节）
权限细节由 RolePermissionCache（进程内内存缓存）提供，不写入 JWT
缓存启动时全量加载 role→permissions + 继承解析，角色/权限变更时自动 reload

部门: 组织架构维度，User ↔ Department 多对多
department_ids 已写入 JWT Claims，供 Phase 2 数据范围过滤使用

Phase 2: DataScope 数据权限
基于 department_ids 实现行级数据隔离：
  销售只看本部门报价单，仓管只操作指定仓库
  实现方式：Service 层查询自动拼接 department_id 过滤条件
  管理员可跨部门查看（super_admin 或特殊角色）
设计原则：RBAC 角色权限模型 | JWT 只存 ID 不存详情 | RolePermissionCache 内存缓存| 路由权限表统一拦截（PERMISSION_ROUTE_MAP） | 资源权限代码注册（16 种资源 × READ/WRITE/DELETE）| super_admin 绕过检查 | 角色层级(parent_role_id)预留 | department_ids 写入 JWT 供 Phase 2 DataScope | 所有变更操作带 operator_id 审计 | AuditLog 记录用户/角色/权限变更

组件间协作关系

DocumentSequence ─── 被所有 8 个模块消费 ─→ 生成编号（33 种 DocumentType | Sequential 策略 + Timestamp 策略）
DocumentLink ──────── 业务流转时写入 ──→ 记录单据间关系链（7 种 LinkType）
InventoryReservation ─ 预留/释放/过期 ──→ 控制库存可用量（4 个模块消费）
CostEntry ──────────── 成本事件触发 ──→ 双层记账统一入口（7 个模块写入，FMS 聚合查询）
DomainEventBus ────── 事件发布/订阅 ──→ 跨模块解耦（19 种事件 | 同步强一致 + Outbox 异步分发 | mark_processed 批量版 | find_events 统一查询）
EventHandlerRegistry ── Handler 注册/分发 ──→ 支持同步（库存预留/质量关卡）和异步（Outbox）两种模式
StateMachineService ─ 状态转换管理 ──→ 统一单据生命周期（替换分散的 if-else 状态校验）
AuditLogService ──── 操作审计追踪 ──→ 业务 Service 直接调用（同事务内，字段级 diff，全链路可追溯）| query_logs 统一查询（支持所有过滤）
IdempotencyService ── 幂等去重保障 ──→ 三级幂等：API防重复提交 + Handler防重复消费 + Saga补偿防重复执行
DeadLetterService ── 死信队列运维 ──→ list_dead_letters 查询 / retry_one 重试 / archive 归档（EventProcessor 超限自动标记 DEAD_LETTER，人工通过此接口干预）
ServiceContext ──────────── 服务调用上下文 ──→ 包装 PgExecutor + operator_id + department_id + data_scope(行级权限) + trace_id + request_id，所有共享服务第一参数
TransactionMode ─────── 显式事务模式声明 ──→ InCallerTx（调用者事务）/ IndependentTx（独立事务）/ AsyncOutbox（异步消费） DomainError ────────── 统一错误模型 ──→ 唯一标准错误类型（ServiceError deprecated），gRPC 语义化 status code
PaginatedResult ──── 统一分页结构 ──→ 所有查询接口返回 { items, total, page, page_size, total_pages }
BatchResult ———————— 批量操作统一返回 —→ 所有 batch_* 方法返回 { success_count, failed_items, total }
UserService ────────── 用户 CRUD + 角色分配 ──→ 创建/更新/删除带 operator_id 审计，batch_assign_roles 批量分配
RoleService ────────── 角色 CRUD + 权限分配 ──→ assign_permissions 接受 (resource_code, action) 元组列表
AuthService ────────── JWT 认证 ──→ login/refresh_token/get_user_claims，Claims 只存 ID（role_ids/department_ids），权限细节由 RolePermissionCache 内存缓存提供
PermissionService ── 权限校验 ──→ check_permission 单条校验 | batch_check_permissions 批量校验（列表接口用，避免 N+1）| 两者均走 user→roles→role_permissions 链路聚合
DepartmentService ── 部门 CRUD + 用户关联 ──→ assign_departments/remove_departments 管理用户-部门多对多

EventBus + StateMachine 构成事件驱动架构骨架，所有业务模块和 Workflow Engine 均通过它们协作
所有共享实体通过 DocumentType 枚举与具体业务模块解耦
ServiceContext + DomainError + TransactionMode + PaginatedResult + EventPublishRequest + AuditAction + SideEffect 构成统一的接口契约基础，ServiceContext 提供事务上下文 + 操作元数据，TransactionMode 显式声明事务边界，新服务直接采用，老服务渐进迁移
