① 统一文档编号 (DocumentSequence)

entity DocumentSequence {
  id: UUID PK
  prefix: String // "SO", "PO", "WO"
  current_value: i32
  date: Date // 按日期分段
  padding_length: i32 // 补零位数
}

enum DocumentType {
  QUOTATION, SALES_ORDER, SHIPPING_REQUEST
  SALES_RETURN, RECONCILIATION
  PURCHASE_QUOTATION, PURCHASE_ORDER
  PURCHASE_RETURN, MISCELLANEOUS_REQUEST
  WORK_ORDER, OUTSOURCING_ORDER
  INSPECTION_REPORT, MRB_REPORT
  RECEIPT, PAYMENT_REQUEST, INVOICE
  }

功能

为所有业务单据生成唯一、可读、有序的编号。格式为 {前缀}-{年}-{月}-{序号}，如 SO-2026-05-00142

解决什么问题

杜绝各模块自行编号导致的编号冲突、格式混乱、无法追溯。按日期分段确保每月序号从 001 重新开始

使用场景

创建任何业务单据时调用：
销售报价 QUO-2026-05-00001
采购订单 PO-2026-05-00032
工单 WO-2026-05-00008
检验报告 INS-2026-05-00015

② 文档关联图谱 (DocumentLink)

entity DocumentLink {
  id: UUID PK
  source_type: DocumentType
  source_id: UUID
  target_type: DocumentType
  target_id: UUID
  link_type: LinkType
  created_at: Timestamp
  created_by: UUID FK→users
}

enum LinkType {
  DERIVED_FROM // 报价→订单
  TRIGGERS // 订单→发货申请
  REFERENCES // 退货→原发货
  RECONCILES // 对账单→发货
  INSPECTS // 检验→来料通知
  FULFILLS // 采购入库→采购单
  ALLOCATES // 领料→工单
  }

功能

以有向图的方式记录任意两个业务单据之间的关系。支持图遍历查询，可追溯一张报价单到最终发票的完整链路

解决什么问题

九大模块间单据关系零散记录在各表里，事后对齐几乎不可能。统一图谱后：一键追溯、BI 报表、ISO 9001 / IATF 16949 合规审计

使用场景

报价 → 销售订单 → 发货申请 → 出库单 → 对账单 → 应收
采购订单 → 来料通知 → IQC 检验 → 入库单
工单 → 领料单 → 报工记录 → FQC → 入库

③ 库存预留层 (InventoryReservation)

entity InventoryReservation {
  id: UUID PK
  product_id: UUID FK→products
  warehouse_id: UUID FK→warehouses
  reserved_qty: Decimal(10,6)
  reservation_type: ReservationType
  source_type: DocumentType
  source_id: UUID
  status: ReservationStatus
  priority: i32 // 预留优先级
  expires_at: Timestamp // TTL自动过期
  created_at: Timestamp
}

enum ReservationType {
  HARD // 工单锁定，不可抢占
  SOFT // 销售预留，可被高优先级抢占
  SAFETY_STOCK // 安全库存永久预留
  }
enum ReservationStatus {
  ACTIVE, FULFILLED, CANCELLED, EXPIRED
  }

功能

区分「现有量」和「可用量」。提供预留、释放、过期自动清理机制。
可用量 = 现有量 − 已预留量

解决什么问题

当前系统只有库存总量，没有可用量（ATP）。导致：销售接单不知道能不能发、采购不知道该买多少、工单领料时才发现库存不够

使用场景

销售订单确认 → SOFT 预留（TTL 7天）
工单下达 → HARD 预留（直到领料或完工）
采购建议计算 = 需求量 − 现有量 − 已预留
安全库存 = 永久预留，不参与可用量计算

④ 成本累积账本 (CostEntry)

entity CostEntry {
  id: UUID PK
  entity_type: CostEntityType
  entity_id: UUID
  cost_type: CostType
  debit_amount: Decimal(10,6)
  credit_amount: Decimal(10,6)
  cost_center: UUID // 部门/产线
  profit_center: UUID
  period: String // "2026-05"
  source_type: DocumentType
  source_id: UUID
  created_at: Timestamp
}

enum CostType {
  MATERIAL, LABOR, OVERHEAD
  OUTSOURCE, REWORK, SCRAP
  }
enum CostEntityType {
  PRODUCT, WORK_ORDER, SALES_ORDER
  PURCHASE_ORDER, INSPECTION
  }

功能

统一成本分类账。每个产生成本的事件写入同一张表，遵循双层记账原则（每个操作同时记借方和贷方）

解决什么问题

传统 ERP 陷阱：运营模块先建、财务后加，事后对账痛苦。从第一天起就记录成本，FMS 报表、利润中心 P&L、毛利分析全部从这一张表查询

使用场景

物料入库 → 记材料成本（借：存货 / 贷：应付）
工序完工 → 记人工+制费（借：在制品 / 贷：人工费）
检验不良 → 记返工/报废成本
外协收货 → 记外协服务成本
新增成本来源（如能源）只需插入行

组件间协作关系

DocumentSequence ─── 被所有模块消费 ───→ 生成单据编号
DocumentLink ──────── 业务流转时写入 ──→ 记录单据间关系链
InventoryReservation ─ 预留/释放/过期 ──→ 控制库存可用量
CostEntry ──────────── 成本事件触发 ──→ 双层记账统一入口

所有共享实体通过 DocumentType 枚举与具体业务模块解耦
