enum ReservationType {
  HARD // 工单锁定，不可抢占
  SOFT // 销售预留，可被高优先级抢占
  SAFETY_STOCK // 安全库存永久预留
  }
enum ReservationStatus {
  ACTIVE, FULFILLED, CANCELLED, EXPIRED
  }

功能

区分「现有量」和「可用量」。提供预留、释放、过期自动清理机制。
可用量 = 现有量 − 已预留量

解决什么问题

当前系统只有库存总量，没有可用量（ATP）。导致：销售接单不知道能不能发、采购不知道该买多少、工单领料时才发现库存不够

使用场景

Sales: 订单确认→SOFT预留（TTL 7天）
MES: 工单下达→HARD预留（直到领料或完工）
Purchase: 到货→释放安全库存预留
WMS: 领料→消耗HARD | 盘点→校准预留
采购建议 = 需求量 − 现有量 − 已预留

⑤ 领域事件总线 (DomainEventBus + Outbox)

entity DomainEvent {
  id: i64 PK BIGSERIAL
  event_type: DomainEventType
  aggregate_type: String // "SalesOrder"
  aggregate_id: i64 // 业务实体 ID
  payload: JSONB // 事件详情
  operator_id: i64
  idempotency_key: String // UNIQUE 防重复
  status: EventStatus // Pending→Processed/Failed
  retry_count: i32 // default 0
  processed_at: Timestamp? // 处理完成时间
  created_at: Timestamp
}

enum EventStatus {
  PENDING // 待处理
  PROCESSING // 处理中
  PROCESSED // 已完成
  FAILED // 失败待重试
  DEAD_LETTER // 超过重试上限
}

trait DomainEventBus {
  publish(event_type, aggregate_type,
    aggregate_id, payload, operator_id): i64
  mark_processed(id): Result
  mark_failed(id, reason): Result
  find_events(aggregate_type, id): Vec
  find_by_type(event_type, since?): Vec
}

trait EventHandlerRegistry {
  register(event_type, handler)
  dispatch(event): Result
  list_handlers(event_type): Vec
}

struct EventProcessor {
  pool: Arc<PgPool>
  registry: EventHandlerRegistry
  max_retries: i32 // default 3
  run() → JoinHandle // 后台 tokio task
  process_event(event): Result
  retry_failed(): u64
}

功能

跨模块解耦的核心机制。Outbox 模式：业务事务内将事件写入 domain_events 表（Outbox），后台 EventProcessor 通过 LISTEN/NOTIFY 实时消费并异步分发给注册的 Handler

解决什么问题

模块间直接调用导致的紧耦合：新增下游逻辑必须改上游 Service。事件驱动后，新逻辑只需注册 handler，上游代码零改动。Outbox 保证：事件与业务数据在同一事务内写入，零丢失风险

19 种事件（按模块）

Sales: SalesOrderConfirmed / Shipped / ReturnReceived
Purchase: POConfirmed / ArrivalReceived / PaymentPaid
MES: PlanReleased / WOReleased / WOCompleted / ReceiptConfirmed
OM: OutsourcingSent / Received / ConvertedToInternal
QMS: InspectionPassed / Failed / MRBDispositioned / RMACreated
FMS: CashJournalConfirmed / WriteOffCompleted

Handler 实现示例（注册到 Registry）

同步强一致：库存预留/释放、质量关卡硬门 — 调用者事务内直接执行
异步 Outbox：成本记录、文档链接、Workflow 触发、通知 — 后台异步消费

InventoryReservationHandler → 预留/释放库存 [同步]
CostEntryHandler → 记录成本分录 [异步]
DocLinkHandler → 创建单据关联 [异步]
WorkflowTriggerHandler → 触发工作流 [异步]
StateTransitionHandler → 触发状态转换 [异步]

Outbox 工作流（异步 Handler）

写入: 业务事务内 INSERT domain_events (status=PENDING) + NOTIFY
消费: EventProcessor 收到 LISTEN → FETCH FOR UPDATE SKIP LOCKED
分发: EventHandlerRegistry.dispatch() → 异步 Handler 执行
完成: 成功 → mark_processed() | 失败 → mark_failed() + 指数退避重试
死信: retry_count > max_retries → DEAD_LETTER（人工介入）

关键约束：Outbox 模式 — 业务事务内写入事件 | EventProcessor 后台异步消费 | LISTEN/NOTIFY 低延迟驱动 | 失败重试（指数退避）+ 死信队列 | idempotency_key 防重复消费 | Handler 独立事务，失败不影响发布方

功能

统一管理所有业务单据的生命周期状态。状态定义和转换规则存 DB 可动态配置，guard 条件复用 Condition AST

解决什么问题

各 Service impl 中分散的 if-else 状态校验，难以维护、容易遗漏。统一状态机后：规则集中管理、变更无需改代码、审计日志自动记录

transition() 校验流程

1. 查询当前状态（from_state）
2. 匹配转换规则（from_state → to_state）
3. 校验 guard_condition（如果存在）
4. 插入 EntityStateLog（追加写）
5. 执行 side_effects（发布事件等）

各模块状态链示例

SalesOrder: Draft → Confirmed → InProduction → Shipped → Completed
WorkOrder: Draft → Planned → Released → InProduction → Completed → Closed
PurchaseOrder: Draft → Confirmed → PartiallyReceived → Received → Closed
OutsourcingOrder: Draft → Sent → InProduction → Delivered → Received → Closed

设计原则：定义存 DB 可动态配置 | guard 复用 Condition AST | EntityStateLog 追加写不可变 | 替换分散的 if-else 状态校验

功能

以追加写方式记录所有业务实体的状态变更。每次 create、update、delete、状态转换都自动写入审计日志，不可修改或删除

解决什么问题

传统 ERP 审计靠人工补记录、查日志，事后无法还原操作轨迹。统一审计后：ISO 9001 合规、财务审计、问题排查均可一键查询

记录来源（业务 Service 直接调用）

核心机制: 业务 Service 在同一事务内直接调用 AuditLogService.record()，携带字段级 diff 和请求上下文
create: 所有 Service 的 create 方法
update: 所有 Service 的 update 方法（含字段级 old→new diff）
transition: StateMachineService.transition() 内调用
delete: 软删除（deleted_at 设置）
approve/reject: Workflow 审批操作

设计原则：业务 Service 直接调用（同事务内，字段级 diff + 请求上下文）| Append-only 不可变 | 按 entity_type 分区索引 | 定期归档到冷存储

⑧ 幂等去重服务 (IdempotencyService)

entity IdempotencyRecord {
  id: i64 PK BIGSERIAL
  idempotency_key: String UNIQUE "{event_id}:{handler}"
  event_id: i64 FK→domain_events
  handler_name: String
  status: String // Processing / Processed
  result: JSONB? // handler 输出（可选）
  created_at: Timestamp
  expires_at: Timestamp? // 过期清理
}

trait IdempotencyService {
  check_and_mark(event_id, handler): bool
  mark_processed(event_id, handler, result?)
  cleanup_expired(before): u64
}

功能

为系统提供 三级幂等保障：API 级（防重复提交）、Handler 级（Outbox 重试幂等）、补偿级（Workflow Saga 重试幂等）。即使同一操作被多次触发，也只执行一次

解决什么问题

Outbox 重试：EventProcessor 崩溃后事件被重新投递 → 防成本记双份、文档链接重复
API 重复提交：前端双击/网络重试 → 防重复创建订单/发货单
Workflow 补偿：Saga 重试时 → 防补偿操作重复执行

工作流

1. EventProcessor 取出事件 → 准备分发给 Handler
2. check_and_mark(event_id, handler_name)
   → true: 首次执行，继续
   → false: 已处理过，跳过
3. Handler 执行
4. mark_processed() 记录结果
5. 定期 cleanup_expired() 清理过期记录

设计原则：三级幂等（API + Handler + Saga补偿）| idempotency_key UNIQUE 原子性 | 与 domain_events.idempotency_key 互补（事件级 vs 操作级）| 过期记录定期清理

组件间协作关系

DocumentSequence ─── 被所有 8 个模块消费 ─→ 生成单据编号（32 种 DocumentType）
DocumentLink ──────── 业务流转时写入 ──→ 记录单据间关系链（7 种 LinkType）
InventoryReservation ─ 预留/释放/过期 ──→ 控制库存可用量（4 个模块消费）
CostEntry ──────────── 成本事件触发 ──→ 双层记账统一入口（7 个模块写入，FMS 聚合查询）
DomainEventBus ────── 事件发布/订阅 ──→ 跨模块解耦（19 种事件 | 同步强一致 + Outbox 异步分发）
EventHandlerRegistry ── Handler 注册/分发 ──→ 支持同步（库存预留/质量关卡）和异步（Outbox）两种模式
StateMachineService ─ 状态转换管理 ──→ 统一单据生命周期（替换分散的 if-else 状态校验）
AuditLogService ──── 操作审计追踪 ──→ 业务 Service 直接调用（同事务内，字段级 diff，全链路可追溯）
IdempotencyService ── 幂等去重保障 ──→ 三级幂等：API防重复提交 + Handler防重复消费 + Saga补偿防重复执行

EventBus + StateMachine 构成事件驱动架构骨架，所有业务模块和 Workflow Engine 均通过它们协作
所有共享实体通过 DocumentType 枚举与具体业务模块解耦
