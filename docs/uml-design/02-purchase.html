SRM 模块 × 共享层交互汇总

DocumentSequence → PQ/PO/PRT/REC/PAY/MISC 编号
DocumentLink → 报价→订单→入库→对账→付款 全链路关联
InventoryReservation → 安全库存预留 → 到货后释放

设计审阅补充说明 (v2)

事件驱动解耦

PurchaseOrder.confirm 前置校验：
  • SupplierService.get() 校验 status ∉ {Blacklisted, Disqualified}
  • 所有 Item quantity > 0 且 unit_price > 0
  • 若关联 PurchaseQuotation，校验 quotation.status == Active 且 valid_until > now
  • 校验通过后才 publish(PurchaseOrderConfirmed) → Outbox → WMS 异步创建 ArrivalNotice
禁止跨模块同步 RPC 调用，所有跨模块交互通过 DomainEventBus + Outbox 模式解耦

状态机流转规则

PurchaseOrderStatus：Draft/Confirmed → Cancelled | Confirmed → PartiallyReceived → Received → Closed
PurchaseReturnStatus：Draft → Confirmed → Shipped → Settled
  Shipped → Settled 由 PurchaseReconciliation.confirm() 异步驱动，或监听 WMS SupplierReturnReceived 事件
  状态变更时发布 PurchaseReturnSettled 事件 → PurchaseReturnSettledHandler 写反向 AP 台账(Debit 冲减应付, Issue #85)
PurchaseReconStatus：Draft → Confirmed → Settled（独立于 Sales ReconciliationStatus）
PurchaseQuotationStatus：Draft → Active（activate 方法触发）→ Expired → Cancelled
MiscRequestStatus：Approved → Purchasing → Received → Closed。如需转正式 PO 在业务层处理，不增加枚举状态

SRM / FMS 边界

SRM 管"应付申请与三单匹配"，FMS 管"资金流水与核销"
PaymentRequest.mark_paid_by_fms() — FMS 执行付款后回调，或监听 PaymentExecuted 事件异步更新
SRM 不直接执行付款动作，只发起申请并接收 FMS 结果

三单匹配校验规则

PaymentRequest.create 校验：PO.received_qty（WMS 同步）+ Invoice qty/amount 必须一致
支持配置 tolerance_rate（如 ±0.5%），超出则阻断创建

退货冲减对账公式

应付总额 = ∑(收货金额) − ∑(退货金额) + 调整项
PurchaseReconItem 新增 returned_qty / returned_amount 字段，退货款自动抵扣

乐观锁与并发控制

所有写操作（confirm, approve, update）底层使用 WHERE id = ? AND updated_at = ?
失败抛出 DomainError::ConcurrentConflict

行级数据权限 (RLS)

list 底层强制过滤 DataScope，禁止前端传参越权
DataScope::Self → WHERE operator_id = ctx.operator_id
DataScope::Department → WHERE department_id IN ctx.allowed_dept_ids

软删除与唯一索引

PurchaseReconciliation unique(supplier_id, period) 使用 Partial Unique Index
CREATE UNIQUE INDEX ... WHERE deleted_at IS NULL

幂等性设计

create, confirm, approve 等写接口支持 idempotency_key
依赖共享层 IdempotencyService，防止网络重试导致重复创建/审批

数据库视图 v_purchase_demands

CREATE VIEW v_purchase_demands AS
  SELECT d.id, d.source_id AS order_id, d.source_line_id AS order_line_id,
         d.product_id, d.required_qty AS quantity, d.required_date, d.priority,
         d.status AS demand_status, d.acquire_channel, d.target_doc_id, d.created_at,
         p.name AS product_name, p.code AS product_code, so.doc_number AS order_no
  FROM demands d
  JOIN products p ON p.id = d.product_id
  JOIN sales_orders so ON so.id = d.source_id
  WHERE d.acquire_channel = 2 AND d.deleted_at IS NULL;

create_order_from_demands 流程

⚠ 事务要求：调用方必须在事务中调用此方法（tx.begin() → 传 &mut tx）。
步骤 1 的乐观锁会把 status 从 Pending→Confirmed，若后续步骤失败需整体回滚，
否则需求成为孤儿（status=Confirmed 但无 target_doc），导致后续采购阻断。

1. 乐观锁：UPDATE demands SET status=Confirmed WHERE id=ANY($1) AND status=Pending AND acquire_channel=2 RETURNING *
2. 计算被跳过的需求（不在 RETURNING 集合中的 demand_ids）
3. 按 product_id 聚合成功锁定的需求（合并数量）
4. 调用 PurchaseOrderService::create 创建 PO 草稿
5. 逐条更新 target_doc (doc_type=PurchaseOrder, doc_id=po_id)
6. 逐条发布 DemandConfirmed 事件（同事务内）
7. 返回 CreateDownstreamResult（含 doc_id、处理数、跳过列表）
