MES Module — UML Class Diagram

核心业务流

Demand 销售需求 → WorkOrder 工单 → ProductionBatch 生产批次（流转卡） → WorkOrderRouting 工序 → WorkReport 报工 → ProductionReceipt 完工入库
↗ OutsourcingOrder 委外单（由 OM 模块管理，MES 通过 OM 接口调用）    ↗ ProductionInspection 生产报检

WorkOrder 工单

扁平化后的生产指令+执行容器：需求直达生成 Draft 工单（定义生产什么/BOM/工艺/数量/排程）；下达后拆批执行由 ProductionBatch 负责

ProductionBatch 生产批次（工序流转卡）

工单按班组/日期拆分为多个批次，每个批次对应一张流转卡（QR 码标签），跟随物料从第一道工序走到最后一道工序

entity ProductionBatch {
  id: i64 PK
  batch_no: String // WO-2026-05-xxxxx-01；= 产成品库存批次号，完工入库时透传 stock_ledger.batch_no
  card_sn: String // 流转卡序列号（二维码唯一标识）
  work_order_id: i64 FK→work_orders
  product_id: i64 // 冗余字段，便于流转卡查询
  batch_qty: Decimal(18,6) // 本批次数量
  completed_qty: Decimal(18,6) // 已完成数量（聚合自 WorkReport）
  scrap_qty: Decimal(18,6) // 报废数量
  team_id: i64? // 分配班组
  current_step: i32 // 当前可开工序序号（状态机核心字段）
  actual_start: Timestamp? // 实际开始时间
  actual_end: Timestamp? // 实际完成时间
  status: BatchStatus
  operator_id: i64
  created_at: Timestamp
  updated_at: Timestamp
}

enum BatchStatus {
  PENDING // 待生产
  IN_PROGRESS // 进行中
  PENDING_RECEIPT // 待入库（最后工序完成后自动触发）
  COMPLETED // 已完成
  CANCELLED
  }
核心规则：
• 执行层实体：跟踪实际生产进度、工序状态机、班组分配
• 工单下达后由项目经理手动创建批次（sum ≤ planned_qty）
• sum(batch_qty) ≤ work_order.planned_qty
• current_step 由系统自动维护，禁止前端直接修改
• 前道工序未完成时拦截下一道报工（防跳序）
• 最后一道工序确认后自动推进至 PendingReceipt
• 批次语义统一：流转卡 batch_no 即产成品库存批次号。完工入库(ProductionReceipt.confirm)时必须透传至 inventory_transactions.batch_no 与 stock_ledger.batch_no，保证产成品库存可按流转卡/工单双向追溯。反冲(backflush)是标准倒扣、不指定具体批次，原材料批次追溯走领料(material_requisition)链路。

BatchStatus 状态流转链

当前状态 目标状态 触发方法 前置条件 副作用
Pending InProgress confirm_routing_step(step=1) current_step == 0 actual_start = now()
InProgress InProgress confirm_routing_step(step=N) current_step == step_no - 1 原子增量 completed_qty
InProgress PendingReceipt advance_to_receipt 最后工序已确认 通知仓管员
PendingReceipt Completed ProductionReceipt.confirm（仓库确认） QMS FQC 硬门 + 仓库指定 入库 + 倒冲 + 释放预留
InProgress Suspended suspend(reason) 可由审批流接管 保留 HARD 预留
Suspended InProgress resume() 可由审批流接管 无
* Cancelled scrap / cancel(WO) 无未领料 / 在途批次 释放 HARD 预留 + 回退 demand（status→Pending，清 target_doc，需求回池可重新规划）

DefectReason 不良原因枚举：
• MaterialDefect 物料不良 — affect_wage=true 照常计工资
• EquipmentFault 设备故障 — affect_wage=true 照常计工资
• OperatorError 操作失误 — affect_wage=false 不良品不计工资
• ProcessIssue 工艺问题 — affect_wage=true 照常计工资
工资公式：(completed_qty + non_operator_defect_qty) × unit_price

设计意图与约束声明

实体归属与快照语义

WorkOrderRouting = 工单级工序快照（Layer 2，定义+参数）
• 自动加载：工单创建时（WorkOrderService::create）从产品 BOM 关联的工艺路线（bom_routings）自动加载工序快照（含产出品/单价/工作中心/委外），只读不可编辑
• 属性源自模板：产出品/单价/工作中心/委外由工艺路线模板（routing_steps）定义，工单只复制，不在工单级编辑（ERPNext/Odoo/OFBiz 共识）
• release 校验非空：下达时校验工序非空；BOM 未关联工艺路线时工序为空，drawer 引导去「工艺路线管理」关联；release 对无工序老工单兜底补加载
• 三层工序模型（已实现）：执行进度迁移至 BatchRoutingProgress
• Layer 1: master_data.routings（工序模板）
• Layer 2: work_order_routings（工单工序快照）
• Layer 3: batch_routing_progress（批次执行进度，写真相源）

ProductionBatch.current_step = 工序状态机核心
• 禁止业务代码直接赋值
• 变更必须通过 confirm_routing_step / advance 触发
• Guard 校验：step_no == current_step + 1（防跳序）
• 同一工序已确认时拒绝重复报工

跨模块契约边界

MES 不持有 OM/QMS 状态，仅消费事件：
• OM → MES: OutsourcingReceived 事件
  handler 推进 BatchRoutingProgress.completed_qty + advance batch
• QMS → MES: FQC 检验结果（同步查询）
  ProductionReceipt.confirm 前硬门校验

审批流边界：
• suspend() / scrap() 为意图接口
• 若需审批：WorkflowEngine 接管
• 审批通过 → 回调 Hook → BatchService 执行状态推进

并发控制策略

① 乐观锁 — WorkOrder.version
  release / close / complete 传入 expected_version
  冲突返回 DomainError::ConcurrentConflict

② SQL 原子增量 — WorkReport 报工
  batch_routing_progress 更新使用:
  SET completed_qty = completed_qty + $delta
  绝不做 read-modify-write

③ 快照语义 — 工艺路线
  WorkOrderRouting 即为快照载体
  工单创建时（create）从 BOM 关联工艺路线自动加载工序快照；BOM 未关联则留空，下达 drawer 引导去关联；release 兜底补加载老工单
  工单级工序只读（属性源自工艺路线模板），报工后锁定；routing_id 记录来源工艺路线

事件边界：同步硬门 vs 异步 Outbox

原则：仅将不影响主流程核心正确性的副作用下沉为异步。库存预留、质量硬门必须同步。

操作 边界 说明
库存 HARD 预留 同步 InvRes.reserve 必须在同一事务内完成
QMS FQC 硬门校验 同步 未通过直接返回 BusinessError 阻断入库
防跳序 Guard 校验 同步 current_step == step_no - 1 在服务层校验
报工数量原子增量 同步 SQL SET qty = qty + delta 在同一事务
成本记录 CostEntry 异步 Outbox 不影响报工正确性，允许短暂延迟
审计日志 AuditLog 同步 InCallerTx 业务操作同事务直接 record，字段级 diff，合规强一致不可丢弃
单据关联 DocumentLink 异步 Outbox WO→PB→WR→PR 关联链可延迟建立
倒冲失败告警 异步 Outbox BackflushShortageEvent → DeadLetter 不阻断入库
委外收货推进工序 异步 Outbox OM OutsourcingReceived → MES handler
Workflow 审批触发 异步 Outbox suspend/scrap 审批由 WorkflowEngine 接管

CREATE VIEW v_production_demands AS
  SELECT d.id, d.source_id AS order_id, d.source_line_id AS order_line_id,
         d.product_id, d.required_qty AS quantity, d.required_date, d.priority,
         d.status AS demand_status, d.acquire_channel, d.target_doc_id, d.created_at,
         p.name AS product_name, p.code AS product_code, so.doc_number AS order_no
  FROM demands d
  JOIN products p ON p.id = d.product_id
  JOIN sales_orders so ON so.id = d.source_id
  WHERE d.acquire_channel = 1 AND d.deleted_at IS NULL;

create_work_orders_from_demands 流程（扁平化：需求直达 Draft 工单）

⚠ 事务要求：调用方必须在事务中调用此方法（tx.begin() → 传 &mut tx）。
步骤 1 的乐观锁会把 status 从 Pending→Confirmed，若后续步骤失败需整体回滚，
否则需求成为孤儿（status=Confirmed 但无 target_doc），导致后续排产阻断。

1. 乐观锁：UPDATE demands SET status=Confirmed WHERE id=ANY($1) AND status=Pending AND acquire_channel=1 RETURNING *
2. 计算被跳过的需求（不在 RETURNING 集合中的 demand_ids）
3. 按 product_id 聚合成功锁定的需求（同物料数量累加，关联首个 SO）
4. 每个聚合组调 WorkOrderService::create 生成 Draft 工单（plan_item_id=None，填 source_so_doc / source_customer）；create 内自动从 BOM 关联工艺路线加载工序（无关联则留空）
5. 逐条更新 target_doc (doc_type=WorkOrder, doc_id=wo_id)
6. 逐条发布 DemandConfirmed 事件（target_doc_type=WorkOrder，同事务内）
7. 返回 CreateWorkOrdersResult（含 wo_ids、处理数、跳过列表）
后续流程：
work-center 工单 card 待下达视图 → 下达 drawer 拆批/排程/加载工序 → WorkOrderService::release
→ 工单完工 → ProductionReceiptService 入库 → DemandService::fulfill → 需求完成
