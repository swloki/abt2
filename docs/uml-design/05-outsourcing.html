Outsourcing Management (OM) — UML Class Diagram

核心业务流

DRAFT ──send()──→ SENT ──receive()──→ RECEIVED ──CLOSED──→ CLOSED
可选中间态（通过追踪节点副作用触发）： SENT ──IN_PRODUCTION──→ DELIVERED ──→ RECEIVED
↗ cancel() → CANCELLED（仅 DRAFT）
↗ convert_to_internal() → CONVERTED_TO_INTERNAL（DRAFT 或 SENT）
架构方向：委外供应商 = WMS 虚拟生产库位，发外协 = WMS 调拨，收回 = WMS 调回

共享基础设施集成

StateMachineService → 所有状态转换（Draft→Sent→Received→Closed 等）统一通过 transition() 管理
AuditLogService → create/send/receive/convert_to_internal 关键操作记录审计日志（同事务）
DomainEventBus → 状态变更后发布领域事件（OutsourcingSent / OutsourcingConvertedToInternal / OutsourcingCancelled；OutsourcingReceived Issue #277 删除，writeback 改同步直调）
DocumentSequenceService → 委外单编号生成（OO-2026-05-xxxxx）
DocumentLinkService → 委外单↔工单 | 委外单↔追踪节点 | 委外单↔调拨单
CostEntryService → 外协成本归集（借:在制品 / 贷:应付外协费）

OutsourcingOrder 委外单

支持全委外/工序委外/材料委外/委外返工四种模式，复用 MES 已有实体

entity OutsourcingOrder {
  id: i64 PK
  doc_number: String // OO-2026-05-xxxxx
  work_order_id: i64? FK→work_orders
  routing_id: i64? FK→work_order_routings
  supplier_id: i64 FK→suppliers
  product_id: i64
  outsourcing_type: OutsourcingType
  planned_qty: Decimal(18,6)
  completed_qty: Decimal(18,6)
  unit_price: Decimal(18,6)
  scheduled_date: Date?
  status: OutsourcingStatus
  virtual_warehouse_id: i64 // WMS 虚拟库位
  source_warehouse_id: i64? // 发料源仓库（send 发料调拨的 from_warehouse）
  version: i32 // 乐观锁，写操作需传 expected_version
  remark: String
  operator_id: i64
  created_at: Timestamp
  updated_at: Timestamp
  deleted_at: Timestamp?
}

enum OutsourcingType {
  FULL // 产品全委外
  PROCESS // 工序委外
  MATERIAL // 材料委外
  REWORK // 委外返工
  }
enum OutsourcingStatus {
  DRAFT, SENT, IN_PRODUCTION
  DELIVERED, RECEIVED, CLOSED
  CONVERTED_TO_INTERNAL // 转自制
  CANCELLED
  }

乐观锁：UPDATE ... SET version = version + 1 WHERE id = $1 AND version = $2，affected_rows == 0 → DomainError::ConcurrentConflict
状态机：所有状态转换通过 StateMachineService.transition() 管理，guard_condition 声明式校验前置条件

OutsourcingMaterial 委外发料明细

追踪每笔委外单的物料发出/回收/在途状态，确保外包资产不流失

entity OutsourcingMaterial {
  id: i64 PK
  outsourcing_id: i64 FK→outsourcing_orders
  product_id: i64 FK→products（物料）
  planned_qty: Decimal(18,6) // 应发数量
  sent_qty: Decimal(18,6) // 已发数量
  returned_qty: Decimal(18,6) // 已收回数量
  unit_cost: Decimal(18,6) // 单位成本
}

计算公式：在途数量 = sent_qty - returned_qty
跨模块：发料/回收通过 WMS.InventoryTransferService 执行
并发控制：父锁代理 — 子表更新在 OutsourcingOrder 的 version 校验事务内完成，子表无独立 version。Repository 层 UPDATE outsourcing_materials ... WHERE outsourcing_id = $1 依赖父表 WHERE version = $2 自然串行化

OutsourcingTracking 委外追踪节点

快递式节点追踪，每个节点有 SLA 时间窗，超时自动升级预警。节点录入需遵循顺序约束。

entity OutsourcingTracking {
  id: i64 PK
  outsourcing_id: i64 FK→outsourcing_orders
  node_type: TrackingNodeType
  tracked_at: DateTime // 实际时间
  planned_at: DateTime? // 计划时间（SLA）
  remark: String?
  operator_id: i64
  created_at: Timestamp
}

enum TrackingNodeType {
  SEND_MATERIAL // 发料（序号 0）
  CARRIER_PICKUP // 承运商揽收（序号 1）
  SUPPLIER_RECEIVED // 供应商接收（序号 2）
  IN_PRODUCTION // 生产中（序号 3）
  SHIPPED // 发货（序号 4）
  IQC_INSPECTED // IQC检验（序号 5）
  WAREHOUSED // 入库（序号 6）
  }

节点流转链路与顺序约束

发料(0) → 承运揽收(1) → 供应商接收(2) → 生产中(3) → 发货(4) → IQC检验(5) → 入库(6)
record_node 校验规则：目标节点的序号必须 > 已完成节点的最大序号
校验函数：validate_node_sequence(completed_nodes, target) → Result<()>
追踪节点为可选增强，不影响核心状态流（SENT 可直接到 RECEIVED）
超时 → 管理台查询 list_overdue，后续可扩展 Worker 定时扫描

接口约定

• 写操作通过 StateMachineService.transition() 管理状态转换，guard_condition 声明式校验前置条件
• 写操作后调用 AuditLogService.record() 记录审计日志（同事务）
• 写操作后调用 EventBus.publish() 发布领域事件（同步 handler 在调用者事务内执行）
• send/receive/convert_to_internal/cancel 需传 expected_version，乐观锁冲突返回 DomainError::ConcurrentConflict
• send/receive/convert_to_internal 内所有操作（状态转换 + WMS调用 + 追踪记录 + 审计 + 事件发布）必须在同一数据库事务内完成，任一步失败整体回滚
• update 和 cancel 仅 DRAFT 状态可用；convert_to_internal 在 DRAFT 或 SENT 状态可用
• 核心状态流：DRAFT → SENT → RECEIVED → CLOSED，IN_PRODUCTION / DELIVERED 为可选中间态（通过追踪节点副作用触发）
• convert_to_internal 返回新工单 ID（InCallerTx 同步创建工单），同时发布 OutsourcingConvertedToInternal 事件
• record_node 内部调用 validate_node_sequence() 校验节点顺序

状态机 Guard 条件（OutsourcingStatus）

StateMachineService.transition() 中配置的 guard_condition 规则，声明式校验状态转换前置条件

转换路径 Guard 条件 业务含义
DRAFT → SENT planned_qty > 0 AND supplier_id IS NOT NULL 必须有数量和供应商
SENT → RECEIVED received_qty > 0 核心路径，可直接收货
SENT → IN_PRODUCTION TrackingNode(SUPPLIER_RECEIVED) exists 可选，追踪节点副作用触发
IN_PRODUCTION → DELIVERED TrackingNode(SHIPPED) exists 可选，追踪节点副作用触发
DELIVERED → RECEIVED received_qty > 0 可选路径的终点
RECEIVED → CLOSED completed_qty >= planned_qty 全部收货完成
DRAFT/SENT → CONVERTED_TO_INTERNAL outsourcing_type IN (FULL, PROCESS) 仅全委外/工序委外可转自制
DRAFT → CANCELLED status == DRAFT 仅草稿可取消
