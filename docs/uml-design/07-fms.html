Financial Management (FMS) — UML Class Diagram v2

业财一体核心流转

销售回款 CashJournal(Inflow) → WriteOff（按出库单核销）→ 自动对应收账款
采购付款 CashJournal(Outflow) → WriteOff（按采购入库单核销）→ 自动对应付账款
费用报销 ExpenseReimbursement → StateMachine（审批流）→ generate_payment_journal → 成本归集
成本核算 CostEntry 统一账本 → 料/工/费完整归集 → 利润中心 P&L
架构方向：统一 cost_entries 分类账，每个产生成本的事件写入同一张表，双层记账 · 财务写操作 InCallerTx + idempotency_key，查询服务 TransactionMode::None

CashJournal 出纳日记账

管控所有现金流入流出：销售回款、采购付款、费用报销、工资支付，钱账实时对齐

entity CashJournal {
  id: i64 PK
  doc_number: String // CJ-2026-05-xxxxx
  journal_type: JournalType
  direction: CashDirection
  amount: Decimal(20,4)
  counterparty_type: CounterpartyType
  counterparty_id: i64
  source_type: DocumentType
  source_id: i64
  bank_account: String
  transaction_date: Date
  period: String // "2026-05"
  status: JournalStatus
  remark: String
  operator_id: i64
  version: i32 // 乐观锁
  created_at: Timestamp
  updated_at: Timestamp
  deleted_at: Timestamp?
}

enum JournalType {
  SALES_RECEIPT // 销售回款
  PURCHASE_PAYMENT // 采购付款
  EXPENSE // 费用报销
  PAYROLL // 工资支付
  OTHER // 其他
  }
enum CashDirection {
  INFLOW // 流入
  OUTFLOW // 流出
  }
enum CounterpartyType {
  CUSTOMER, SUPPLIER, EMPLOYEE, OTHER
  }
enum JournalStatus {
  DRAFT, CONFIRMED, CANCELLED
  }

Rust 模型层：counterparty_type + counterparty_id → CounterpartyRef 枚举（Repository 层拆包/装包，DB 层不变）

CashJournalLine 日记账明细

双层记账：每笔日记账同时记借方和贷方，支持成本中心和利润中心归集

entity CashJournalLine {
  id: i64 PK
  journal_id: i64 FK→cash_journals
  account_code: String // 会计科目编码
  debit_amount: Decimal(20,4) // 借方金额
  credit_amount: Decimal(20,4) // 贷方金额
  cost_center: i64? // 成本中心
  profit_center: i64? // 利润中心
  remark: String
}

记账规则：每笔 CashJournal 至少两条明细行（一借一贷），借方总额 = 贷方总额

WriteOff 核销

销售回款按出库单核销，采购付款按入库单核销，确保每笔钱都有据可查

entity WriteOff {
  id: i64 PK
  write_off_type: WriteOffType
  cash_journal_id: i64 FK→cash_journals
  source_type: DocumentType
  source_id: i64 // 出库单/入库单
  amount: Decimal(20,4)
  write_off_date: Date
  operator_id: i64
  created_at: Timestamp
}

enum WriteOffType {
  SALES_RECEIPT // 销售回款核销
  PURCHASE_PAYMENT // 采购付款核销
  }

核销规则：销售回款按出库单逐笔核销，确保每笔进账精准对齐到具体客户和发货批次

P0 防超核销：FOR UPDATE 锁源单据可用余额 → 校验 unreconciled ≥ amount → DB CHECK 约束兜底 → 超核销抛 DomainError::BusinessRule

幂等策略：write_offs 表唯一索引 uk_write_offs_idempotency ON (idempotency_key) WHERE idempotency_key IS NOT NULL，无需调用 IdempotencyService

员工费用报销申请，审批通过后由 WorkflowEngine AutoAction 触发 generate_payment_journal 内部创建付款日记账

entity ExpenseReimbursementItem {
  id: i64 PK
  reimbursement_id: i64 FK→expense_reimbursements
  expense_type: ExpenseType
  amount: Decimal(20,4)
  description: String
  receipt_no: String?
  cost_center: i64?
  profit_center: i64?

enum ExpenseType {
  TRAVEL, OFFICE, TRANSPORT, MEAL, OTHER
  }
enum ExpenseStatus {
  DRAFT, SUBMITTED, APPROVED, PAID, CANCELLED
  }

工作流解耦：submit/approve 由 StateMachineService.transition() 驱动，FMS 只负责领域操作（create/get/list + generate_payment_journal）

AutoAction：generate_payment_journal 由 WorkflowEngine Hook 调用（禁止前端直调），内部创建 CashJournal，失败返回 Result 由 Hook 重试/死信

Saga 补偿：失败回滚已创建 CashJournal · Expense 保持 Approved 等待重试 · 超过重试上限进死信队列人工介入

状态推进：Draft→Submitted→Approved 由 StateMachineService 驱动 · Approved→Paid 由 generate_payment_journal 内部更新 · 禁止外部直接修改 status · 乐观锁 version 防并发冲突

Service 接口

interface CashJournalService {
  create(ctx, req): Result<i64> [InCallerTx]
  confirm(ctx, id, idempotency_key?): Result<()> [InCallerTx]
    // 锁定→聚合借贷→校验→状态变更→审计→事件，同一 COMMIT
  get(ctx, id): Result<CashJournal>
  list(ctx, filter: CashJournalFilter, page): PaginatedResult
  get_balance(ctx, period): Result<BalanceSummary>
}

interface WriteOffService {
  write_off(ctx, req: WriteOffReq): Result<i64> [InCallerTx]
    // FOR UPDATE 锁源单据 → 校验未核销余额 → DB CHECK 兜底
  list_by_source(ctx, source_type, source_id, page): PaginatedResult
  get_unreconciled_amount(ctx, source_type, source_id): Result<Decimal> // 只读快照
}
interface ExpenseReimbursementService {
  create(ctx, req): Result<i64> [InCallerTx]
  get(ctx, id): Result<ExpenseReimbursement>
  list(ctx, filter: ExpenseFilter, page): PaginatedResult
  generate_payment_journal(ctx, expense_id): Result<i64> [IndependentTx] internal
}
// Draft→Submitted→Approved: StateMachineService.transition()
// Approved→Paid: generate_payment_journal 内部更新
// 禁止外部模块直接修改 ExpenseReimbursement.status
// Hook 异步触发，独立事务，失败返回 Result 由 Hook 重试/死信

interface CostAccountingService {
  get_product_cost(ctx, product_id, period): Result<ProductCostSummary> [None]
  get_work_order_cost(ctx, work_order_id): Result<WorkOrderCostSummary> [None]
  get_profit_center_summary(ctx, profit_center_id, from, to, page): PaginatedResult [None]
  get_margin_analysis(ctx, order_id): Result<MarginAnalysis> [None]
}
// READ COMMITTED 隔离，最终一致派生值，非实时强一致
// 高频查询建议物化视图或缓存 (TTL=5min)

FMS 模块 × 跨模块交互汇总

Sales CRM → 销售回款 CashJournal(Inflow) + WriteOff 核销到 ShippingRequest
Purchase SRM → 采购付款 CashJournal(Outflow) + WriteOff 核销到 PurchaseOrder/ArrivalNotice
CostEntry 消费 → get_product_cost 汇总料/工/费 | get_work_order_cost 归集到工单
利润中心 → get_profit_center_summary 按部门归集 P&L | get_margin_analysis 报价-实际毛利差异
MES → 完工入库触发成品成本 | 报工触发人工成本 | 倒冲差异触发损耗成本
OM → 外协收货触发外协成本
QMS → 不良处置触发返工/报废成本
Workflow → StateMachineService 驱动报销审批流 → Approval Hook 触发 generate_payment_journal（Saga 补偿）
共享层集成 → DocSeq(CashJournal/ExpenseReimbursement) | AuditLog 同事务 | DomainEventBus 异步 | IdempotencyService 防重复

领域事件（注册到 DomainEventType）

CashJournalConfirmed — 日记账确认后触发下游对账与成本归集
WriteOffCompleted — 核销完成触发应收/应付更新
ExpensePaymentGenerated — 报销付款日记账生成后触发状态流转为 Paid

错误映射（DomainError → gRPC Status）

超核销 → DomainError::BusinessRule("OverWriteOff")
借贷不平 → DomainError::BusinessRule("UnbalancedEntry")
重复提交 → DomainError::Duplicate("IdempotencyKey")
前置状态不满足 → DomainError::BusinessRule("InvalidState")
