Workflow Engine V2 — UML Class Diagram

V2 架构分层

┌─────────────────────────────────────────────────────────────┐
│ Shared Infrastructure（定义于 00-shared-infrastructure.puml）│
│  DomainEventBus + DomainEventHandler + 19 种事件类型  │
│  StateMachineService + 状态定义/转换规则/变更日志      │
└──────────────────────────┬──────────────────────────────────┘
                       │ 依赖
┌──────────────────────────▼──────────────────────────────────┐
│ Workflow Engine（定义于 08-workflow-engine.puml）            │
│  WorkflowCore: 9 种节点类型 | Graph | Condition AST     │
│  Integration: ActionRegistry | HookRegistry | Worker     │
│  Saga: compensate_action | compensation_log | 逆序回滚    │
└─────────────────────────────────────────────────────────────┘

模块边界：DomainEventBus / StateMachineService / DomainEventHandler 属于共享基础设施层，通过 ServiceContext 传递事务上下文和操作元数据，
所有业务模块（Sales、Purchase、WMS、MES...）均可直接使用，不依赖 Workflow Engine。
Workflow Engine 是这些共享服务的消费者，通过 DomainEventBus 发布事件、通过 StateMachineService 触发状态转换。

集成模式对比

V1 — 同步直接调用（紧耦合）

SalesOrderService.confirm()
  ├── update status in DB
  ├── InvRes.reserve()    // 直接调用
  ├── CostEntry.create()  // 直接调用
  └── DocLink.create_link() // 直接调用

问题: Service impl 耦合了其他模块知识
新增逻辑必须改 SalesOrderService

V2 — 事件驱动（松耦合）

SalesOrderService.confirm()
  ├── StateMachine.transition("Confirmed")
  ├── update status in DB
  └── EventBus.publish(EventPublishRequest{ SalesOrderConfirmed })
      ├── InventoryHandler → InvRes.reserve()
      ├── CostEntryHandler → CostEntry.create()
      └── DocLinkHandler → DocLink.create_link()

好处: SalesOrderService 只关心核心逻辑
其他模块通过 handler 自动响应

Workflow Core — 四张核心实体表

引擎基础，V1 已实现，V2 增强 WorkflowInstance 和 WorkflowNode

WorkflowTemplate 流程模板

定义一种业务流程的结构（有向图），一个 entity_type 只能有一个 active 模板

entity WorkflowTemplate {
  id: i64 PK BIGSERIAL
  entity_type: String // "sales_order" | "work_order" | ...
  name: String // "销售订单审批流"
  version: i32 // 每次发布 +1
  status: TemplateStatus // draft | active | archived
  graph: JSONB // WorkflowGraph 序列化
  graph_checksum: String? // 发布时计算的哈希
  trigger_event: String? // 可选: 绑定 DomainEventType
  created_at, updated_at, deleted_at
}

enum TemplateStatus {
  Draft → Active → Archived
  // 只有 Active 的模板可以被实例化
  // publish_template() 时 Draft → Active
}

生命周期：创建为 Draft → publish_template() 经 GraphLinter 校验 → Active
版本控制：每次 publish version+1，checksum 用于变更检测

WorkflowInstance 流程实例

模板的一次运行，frozen_graph 冻结启动时的图结构（不受后续模板修改影响）

entity WorkflowInstance {
  id: i64 PK
  template_id: i64 FK → templates
  template_version: i32? // 启动时的模板版本
  entity_type: String // "sales_order"
  entity_id: i64 // 关联的业务实体 ID
  status: InstanceStatus
  frozen_graph: JSONB // 启动时 snapshot
  context: JSONB // {entity_snapshot, variables, join_progress}
  suspended_reason: JSONB?
  initiator_id: i64 // 启动人
  + parent_instance_id: i64? // SubProcess 父实例
  + saga_compensation_log: JSONB? // Saga 补偿日志
  created_at, updated_at, completed_at
}

context 结构：{entity_snapshot: {...}, variables: {...}, join_progress: {...}}
frozen_graph：启动时从 template.graph 复制，后续模板变更不影响运行中实例

WorkflowTask 任务

每个 Approval 节点创建一个 Task，等待人工审批；AutoTask/ServiceTask 不创建 Task（直接执行并推进）

entity WorkflowTask {
  id: i64 PK
  instance_id: i64 FK → instances
  node_id: String // 对应 frozen_graph 中的节点 ID
  prev_task_id: i64? // 前驱任务（delegate/reject 时链式追溯）
  assignee_id: i64? // 当前审批人
  status: TaskStatus
  action: String? // "approve" | "reject" | "delegate"
  timeout_action: String? // "auto_approve" | "auto_reject" | "notify"
  due_at: DateTime? // 超时截止时间
  remind_at: DateTime? // 提醒时间
  result: JSONB? // {"comment": "同意"}
  created_at, completed_at
}

entity WorkflowHistory {

状态枚举 — 状态机转换规则

InstanceStatus 实例状态

Running ──────────────┐
  ├→ Completed   // 到达 End 节点
  ├→ Rejected    // reject(terminate)
  ├→ Suspended   // auto_task 失败
  ├→ Cancelled   // 手动取消
  ├→ Terminated  // 系统终止
  ├→ Compensating// Saga 补偿中
  └→ Compensated // Saga 已补偿

TaskStatus 任务状态

Pending // 等待审批
  ├→ Completed  // approve
  ├→ Rejected   // reject
  ├→ Delegated  // 委派给他人
  ├→ TimedOut   // Worker 超时处理
  └→ Cancelled  // 实例取消时批量取消

TemplateStatus 模板状态

Draft // 编辑中
  └→ Active // publish_template()
      └→ Archived // archive_template()

规则: 只有 Active 可以实例化
规则: publish 时 GraphLinter 校验
规则: ActionRegistry 验证 action 已注册

NodeType — 9 种节点类型详解

NodeType创建 Task?config 必填说明
Start否无流程起点，有且仅有一个
End否无到达时 instance → Completed
Approval是fallback_assignee, multi_approval, reject_action, timeout_hours, timeout_action等待人工审批，创建 pending Task
AutoTask否action, input_mapping, output_mapping通过 ActionRegistry 执行已注册的动作
ServiceTask否service直接调用指定 Service 方法，类型安全
Join否无特殊汇聚并行分支，所有入边源节点必须有 completed Task
SubProcess否sub_entity_type, wait_for_completion启动子 WorkflowInstance，通过 parent_instance_id 关联
TimerWait否duration_hours 或 until暂停等待时间，Worker 扫描到期后自动推进
SignalCatch否signal暂停等待外部信号，WorkflowService.signal() 触发推进

Approval 节点 config 详解

// 完整 config 示例
{
  "assignee_type": "role",
  "assignee_value": "manager",
  "fallback_assignee": 42, // 必填! assignee 解析失败时兜底
  "multi_approval": "any", // "any" | "all"
  "reject_action": "terminate", // "terminate" | "back_to_previous"
  "timeout_hours": 48,
  "timeout_action": "auto_approve", // "auto_approve"|"auto_reject"|"notify"
  "remind_hours_before": 4
}
multi_approval 逻辑:
"any" = 任意一人 approve → 推进（默认）
"all" = 所有人 approve → 才推进
  一人 approve 后检查剩余 pending 数
  remaining > 0 → 记录 MULTI_APPROVAL_WAITING
  remaining = 0 → 推进到下一节点

reject_action 逻辑:
"terminate" = instance → Rejected
"back_to_previous" = 退回上一审批节点
  找入边的源节点，重新创建 pending Task
  同时 cancel 同 node_id 其余 pending

引擎支撑组件

ActionRegistry + AutoAction

注册业务动作，AutoTask 节点通过 config.action 名称查找执行

trait AutoAction {
  execute(tx, inputs): Result<ActionOutput>
  compensate(tx, inputs): Result<()> // V2
}

struct ActionDef { // 元数据
  name, label, description
  inputs: Vec<FieldDef>
  outputs: Vec<FieldDef>
}

class ActionRegistry {
  register(def, action)
  get(name) → Arc<dyn AutoAction>
  is_registered(name) → bool
  list_defs() → &[ActionDef]
  validate_startup(templates) // 启动校验
}

// 表达式映射引擎
// input_mapping: {"product_id": "${entity_snapshot.product_id}"}
// output_mapping: {"order_id": "production_order_id"}
// resolve_mapping() 解析 ${path.to.field} 语法

启动校验：所有 active 模板引用的 action 必须已注册，否则 panic 拒绝启动（fail-closed）

WorkflowHook 事务后回调

实例完成后（approved/rejected）在事务 commit 之后异步触发的业务回调

trait WorkflowHook {
  on_approved(pool, instance): Result<()>
  on_rejected(pool, instance): Result<()>
}

class HookRegistry {
  register(entity_type, hook)
  get(entity_type) → Option<Arc<Hook>>
}

执行时机:
  approve_task() → tx.commit() 成功后
  → fire_hook("approved")
  → hook.on_approved(pool, instance)

失败处理:
  hook 失败 → 记录 history(hook_executed, {success:false})
  → retry_failed_hook() 可手动重试
  // 不影响已提交的事务

与 DomainEventBus 的区别：Hook 是事务后异步回调，用于非关键操作（如通知）；EventBus 是事务内同步分发，用于关键业务操作。

WorkflowWorker 后台扫描

tokio 后台任务，CancellationToken 控制生命周期

class WorkflowWorker {
  pool: Arc<PgPool>
  cancel_token: CancellationToken
  scan_interval_secs: u64 // 默认 60s
}

每轮扫描:
  scan_overdue_tasks()
  ├── find_overdue_pending_tx(FOR UPDATE SKIP LOCKED)
  ├── timeout_action == "auto_approve" → 完成+推进
  ├── timeout_action == "auto_reject" → 拒绝+推进
  └── 其他 → 仅记录 history

  scan_remindable_tasks()
  ├── find_remindable_pending_tx
  ├── 记录 history(reminder)
  └── clear_remind_at（防止重复提醒）

GraphLinter 发布前校验

publish_template() 时调用，纯函数，不依赖数据库

lint_graph(graph, action_lookup)

校验规则:
  R1 有且仅有一个 Start 节点
  R2 至少一个 End 节点
  R3 边引用的节点必须存在
  R4 DFS 环检测（不允许环）
  R5 Approval 必须有 fallback_assignee
  R6 AutoTask action 必须已注册
  R7 Join 入边 ≥ 2
  R8 back_to_previous 只能在单一入边的 Approval
  R9 ServiceTask 必须有 service config
  R10 SubProcess 必须有 sub_entity_type
  R11 TimerWait 必须有 duration 或 until
  R12 SignalCatch 必须有 signal
  R13 compensate_action 必须已注册

节点处理流程 — process_next_node()

引擎的核心递归函数：从 Start 节点出发，沿边遍历，根据节点类型执行不同逻辑

process_next_node(tx, instance_id, graph, node_id, condition)

  // 1. 条件判断
  if condition → evaluate_condition(cond, context)
  └── 条件不满足 → return Ok(())（不推进）

  // 2. 按节点类型处理
  match node.node_type:
    Start → error（Start 只能是初始节点）
    End → instance.status = Completed
    Approval → 创建 WorkflowTask(Pending, assignee, due_at, remind_at)
    AutoTask → 执行 action_registry.get(action).execute(tx, inputs)
      └── 递归 process_next_node 所有出边
    ServiceTask → 调用 service 方法
      └── 递归 process_next_node 所有出边
    SubProcess → start_instance(sub_entity_type) + 设置 parent_instance_id
      └── wait_for_completion=true → 暂停，等待子实例完成信号
    TimerWait → 记录 expected_fire_at，不推进
      └── Worker 扫描到期后推进
    SignalCatch → 记录等待的 signal_name，不推进
      └── WorkflowService.signal() 匹配后推进
    Join → 检查所有入边源节点是否有 completed Task
      ├── 全部完成 → 递归 process_next_node 所有出边
      └── 未全部完成 → 记录 JOIN_WAITING，等待

迁移策略

Phase 1: 新增基础设施（不改现有代码）
  → 新增 domain_events / state_* / entity_state_logs 表
  → 实现 DomainEventBus + StateMachineService
  → 所有现有功能不受影响

Phase 2: 新模块从第一天使用新架构
  → Sales/Purchase/MES 等新模块直接用 EventBus + StateMachine
  → 注册业务 handler

Phase 3: 逐步迁移现有模块
  → 逐个将 Service impl 中的直接调用改为 EventBus.publish()
  → 每次迁移一个事件类型，确保不退化

Phase 4: 引入 Saga（可选）
  → 为关键长流程定义 Saga 模板
  → 在 ActionRegistry 注册 compensate_action
