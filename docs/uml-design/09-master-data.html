Master Data Module — UML Class Diagram

核心数据流

Category 分类（树形 + 物化路径） → Product 产品（M:N 多分类） → Bom 物料清单（草稿/发布） → BomNode 节点（树形结构）
↗ BomSnapshot 发布快照（不可变，工单引用）    ↗ PriceLogEntry 价格历史（支持时间点查询）    ↗ BomCostReport BOM 成本报告（支持 as_of_date）
Customer 客户（AR 主体，信用管控） → CustomerContact 联系人 → CustomerAddress 收货地址    Supplier 供应商（AP 主体，采购源头） → SupplierContact 联系人 → SupplierBankAccount 银行账户

v4 设计改进

① 客商主数据统一 — Customer 和 Supplier 统一归入主数据模块，避免 Sales/Purchase/FMS 间网状依赖，形成星型拓扑
② ServiceContext — 所有 Service 方法统一使用 ServiceContext（定义见共享基础设施层），与其他模块对齐
③ BomService 拆分 — 按 CQRS 风格拆为 BomQueryService / BomCommandService / BomNodeService / BomCostService 四个 trait
④ 强类型请求结构体 — CreateProductReq / UpdateProductReq / CreateBomReq / NewBomNode 等专用请求类型，区分必填与可选字段
⑤ 乐观锁 — BomCommandService.update / BomNodeService.update_node 要求传入 expected_version，冲突返回 DomainError::ConcurrentConflict。新增 BOM 时 version = 1
⑥ 命名统一 — find → get（单 ID 查询），query → list（分页查询），publish 返回 BomSnapshot
⑦ 领域错误 — 使用 DomainError（定义见共享基础设施层），核心变体：NotFound / ConcurrentConflict / BusinessRule / InvalidStateTransition

Customer 客户

客户主档案，AR（应收账款）主体。支持公海/私海客户池、信用额度管控、多收货地址。owner_id=null 为公海客户（全员可见），有值为私海（归属业务员+上级可见）。下游模块：Sales、FMS、WMS

entity Customer {
  id: i64 PK
  code: String // 客户编码，自动生成
  name: String
  short_name: String?
  category: CustomerCategory
  status: CustomerStatus
  tax_number: String? // 统一社会信用代码
  invoice_title: String? // 开票抬头
  credit_limit: Decimal(20,4)? // 信用额度
  payment_terms: String? // 账期条件
  receivable_account: String? // 应收科目编码
  owner_id: i64? // 客户归属业务员，null=公海客户
  department_id: i64? // 归属部门，支持部门级共享
  remark: String
  operator_id: i64 FK→users
  created_at: Timestamp
  updated_at: Timestamp
  deleted_at: Timestamp?
}

entity CustomerContact {
  id: i64 PK
  customer_id: i64 FK→customers
  name: String
  position: String?
  phone: String?
  email: String?
  is_primary: bool

  id: i64 PK

  address_type: String // Shipping/Billing

  PROSPECTIVE // 潜在客户

  INACTIVE // 停用
  BLACKLISTED // 黑名单

  create(ctx, req) → i64
    // 内部调用 DocumentSequenceService.next_number(ctx, "CUS")

Product 产品

核心基础实体，所有业务模块（库存/BOM/采购/销售/MES）的物料主数据。支持生命周期状态管理和外部编码

entity Product {
  product_id: i64 PK
  pdt_name: String // 产品名称
  product_code: String // 自动生成
  unit: String // 计量单位
  status: ProductStatus // Active/Inactive/Obsolete
  external_code: String? // 客户/供应商编码
  owner_department_id: i64? // 归属部门
  meta: JSONB → ProductMeta
}

enum ProductStatus {
  Active // 在用，出现在下拉框
  Inactive // 停用，不出现在下拉框
  Obsolete // 作废，仅供历史查询
}

struct ProductMeta {
  specification: String // 规格型号
  acquire_channel: String // 获取途径(采购/自制)
  old_code: String? // 旧编码(迁移兼容)
}

entity ProductCategory {
  product_id: i64 FK→products
  category_id: i64 FK→categories
  // UNIQUE(product_id, category_id)
}

struct CreateProductReq {
  name: String // 必填
  unit: String // 必填
  status: ProductStatus
  external_code: String?
  owner_department_id: i64?
  meta: ProductMeta
}

struct UpdateProductReq {
  name: String? // 全部可选
  unit: String?
  external_code: String?
  owner_department_id: i64?
  meta: ProductMeta?
}

trait ProductService {
  // ctx: ServiceContext
  create(ctx, req: CreateProductReq) → Product
    // 内部调用 DocumentSequenceService.next_number(ctx, "PRODUCT")
  update(ctx, id, req: UpdateProductReq) → Product
  delete(ctx, id) // 内部调用 check_product_usage
  get(ctx, id) → Product // 不存在返回 DomainError::NotFound
  get_by_ids(ctx, ids) → Vec<Product>
  list(ctx, filter, page) → PaginatedResult
  check_product_usage(ctx, id, query) → PaginatedResult
}

Category 产品分类

产品分类树，支持无限层级。通过 ProductCategory 关联表实现产品多分类（一个产品可归属多个分类）。物化路径(path)优化子树查询性能

entity Category {
  category_id: i64 PK
  category_name: String // 分类名称
  parent_id: i64 // 父分类(0=顶级)
  path: String // 物化路径 如 "/1/5/12/"
  meta: JSONB → CategoryMeta
  created_at: DateTime
  updated_at: DateTime?
}

struct CategoryMeta {
  count: i64 // 关联产品数量
}

struct CategoryTree {
  category_id, category_name
  parent_id, path, meta
  children: Vec<CategoryTree>
}

trait CategoryService {
  // ctx: ServiceContext
  create(ctx, req) → i64
  update(ctx, category_id, req)
  delete(ctx, category_id) // 内部检查子分类和关联产品
  get(ctx, category_id) → Category
  list(ctx, filter, page) → PaginatedResult
  get_tree(ctx, root_id?,
    depth_limit?) → Vec<CategoryTree>
    // null=全树, id=子树
    // depth_limit 防止全量树爆炸
  move_to(ctx, category_id, new_parent_id)
    // 级联更新path (CTE)
  assign_products(ctx, category_id,
    product_ids)
  remove_products(ctx, category_id,
    product_ids)
}

BOM 物料清单

定义产品的物料构成。草稿/发布双状态，发布时生成不可变快照（BomSnapshot）。支持复制、物料替换、成本核算。version 字段用于乐观锁并发控制

entity Bom {
  bom_id: i64 PK
  bom_name: String
  create_at: DateTime
  update_at: DateTime?
  bom_detail: BomDetail // 从 bom_nodes 加载
  bom_category_id: i64? FK→bom_categories
  status: BomStatus
  version: i32 // 乐观锁版本号
  published_at: DateTime?
  created_by: i64? FK→users
}

enum BomStatus {
  DRAFT // 仅创建者可见
  PUBLISHED // 全员可见

乐观锁与发布流程

BOM 更新和节点更新要求客户端传入 expected_version，冲突返回 ConcurrentConflict。publish 返回 BomSnapshot（不可变），语义更准确

// 乐观锁机制
// 新增 BOM 时 version = 1
// UPDATE boms SET ..., version = version + 1
// WHERE bom_id = $1 AND version = $2
// affected_rows == 0 → DomainError::ConcurrentConflict
// Handler 映射为 gRPC Aborted (ABORTED)
// 服务端不自动重试，客户端获取最新版本后重新提交

// 发布流程（BomCommandService.publish）
publish(ctx, bom_id) → BomSnapshot
  Draft → Published
  // version 自增
  // 自动生成 BomSnapshot(bom_id, version, ...)
  // 同事务内:
  // AuditLog.record(ctx, Publish, ...)
  // EventBus.publish(ctx, BomPublished{...})
  // 返回 BomSnapshot（不可变）

unpublish(ctx, bom_id) → Bom
  Published → Draft
  // 不删除快照

// 物料替换（InCallerTx + FOR UPDATE）
substitute_product(ctx,
  req: SubstituteReq)
  → SubstitutionResult {
    affected_boms, affected_nodes
  }

状态转换规则

ProductStatus
  Active ↔ Inactive — 双向，停用/重新启用
  Active → Obsolete — 作废，不可逆
  Inactive → Obsolete — 作废，不可逆
  Obsolete → * — 不允许任何转换

BomStatus
  Draft → Published — 发布，生成快照，version 自增
  Published → Draft — 取消发布，不删除快照

CustomerStatus
  Prospective → Active — 潜在客户转正
  Active ↔ Inactive — 停用/重新启用
  Active → Blacklisted — 拉黑，发布 CustomerBlacklisted 事件
  Blacklisted → * — 不允许自动恢复，需人工审批

SupplierStatus
  Prospective → Qualified — 准入通过
  Qualified ↔ Probation — 试用期降级/恢复
  Qualified → Disqualified — 取消资格
  * → Blacklisted — 拉黑，发布 SupplierBlacklisted 事件
  Blacklisted → * — 不允许自动恢复，需人工审批

业务校验规则

CustomerService.create
  Validation: code 全局唯一（含软删除行，见 DBA Note）
  Validation: 若传入 tax_number，跨表检索 Customer + Supplier。若已存在，不阻断但返回 Warning："该税号已在系统中存在客商档案 [code: XXX]"，防止重复建档。Warning 记录至 AuditLog

SupplierService.create
  Validation: code 全局唯一（含软删除行，见 DBA Note）
  Validation: 若传入 tax_number，跨表检索 Customer + Supplier。若已存在，不阻断但返回 Warning，同上

CustomerService.validate_contact_ownership
  Sales 模块创建报价单/订单时，校验 contact_id 是否归属 customer_id，防止张冠李戴。内部实现：list_contacts → 内存比对

CustomerService.claim(ctx, id)
  公海客户认领：owner_id=null → owner_id=ctx.operator_id, department_id=ctx.department_id
  Validation: 客户必须为公海状态（owner_id IS NULL）且 status=Active
  Validation: 同一业务员认领上限可配置（防止囤积客户）

CustomerService.transfer(ctx, id, new_owner_id, new_department_id)
  客户转移（高权限，需销售总监角色）
  发布 CustomerTransferred 事件，通知原业务员和新业务员

CustomerService.list — 行级数据过滤（Row-Level Security）
  禁止依赖前端传参过滤，必须根据 ctx.data_scope 在 Repository 层强制拼接 WHERE：
    DataScope::All → 无额外过滤
    DataScope::Department → WHERE department_id = ctx.department_id
    DataScope::Self → WHERE owner_id = ctx.operator_id
  公海客户（owner_id IS NULL）对所有人可见

🚨 银行账户变更风控（P0 高危）

风险说明：供应商银行账户变更是 ERP 排名前三的高危欺诈风险点（邮箱篡改/内部回扣）

SupplierService.add_bank_account / update_bank_account：
  1. 强制 AuditLog — 必须记录操作人 IP、修改前后完整对比（字段级 diff）
  2. 事件发布 — 发布 SupplierBankAccountChanged 事件

FMS 模块监听 SupplierBankAccountChanged：
  3. 7天拦截规则 — 若该供应商在 7 天内发生过银行账户变更，FMS 发起付款时必须强制触发主管二次复核/短信验证，否则拒绝打款
  4. AuditLog 记录 — 复核过程独立留痕，防止内部合谋

DBA Note — 软删除与唯一索引

customers.code / suppliers.code 唯一索引：
  必须使用 Partial Unique Index，兼容软删除：
  CREATE UNIQUE INDEX uk_customers_code ON customers(code) WHERE deleted_at IS NULL;
  CREATE UNIQUE INDEX uk_suppliers_code ON suppliers(code) WHERE deleted_at IS NULL;

tax_number 跨表查询优化：
  tax_number 不建跨表唯一约束（允许客户和供应商共用同一税号），但建议建单表索引加速查重：
  CREATE INDEX idx_customers_tax ON customers(tax_number) WHERE tax_number IS NOT NULL AND deleted_at IS NULL;
  CREATE INDEX idx_suppliers_tax ON suppliers(tax_number) WHERE tax_number IS NOT NULL AND deleted_at IS NULL;
