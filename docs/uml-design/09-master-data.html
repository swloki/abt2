Master Data Module — UML Class Diagram

核心数据流

Category 分类（树形 + 物化路径） → Product 产品（M:N 多分类） → Bom 物料清单（草稿/发布） → BomNode 节点（树形结构）
↗ BomSnapshot 发布快照（不可变，工单引用）    ↗ PriceLogEntry 价格历史（支持时间点查询）    ↗ BomCostReport BOM 成本报告（支持 as_of_date）
LaborProcessDict 工序字典（全局主数据） → Routing 工艺路线模板 → BomRouting 路线绑定（产品↔路线） → MES WorkOrderRouting 运行时快照    BomLaborProcess 劳务工序 → BomCostReport.labor_costs
Customer 客户（AR 主体，信用管控） → CustomerContact 联系人 → CustomerAddress 收货地址    Supplier 供应商（AP 主体，采购源头） → SupplierContact 联系人 → SupplierBankAccount 银行账户

v4 设计改进

① 客商主数据统一 — Customer 和 Supplier 统一归入主数据模块，避免 Sales/Purchase/FMS 间网状依赖，形成星型拓扑
② ServiceContext — 所有 Service 方法统一使用 ServiceContext（定义见共享基础设施层），与其他模块对齐
③ BomService 拆分 — 按 CQRS 风格拆为 BomQueryService / BomCommandService / BomNodeService / BomCostService 四个 trait
④ 强类型请求结构体 — CreateProductReq / UpdateProductReq / CreateBomReq / NewBomNode 等专用请求类型，区分必填与可选字段
⑤ 乐观锁 — BomCommandService.update / BomNodeService.update_node 要求传入 expected_version，冲突返回 DomainError::ConcurrentConflict。新增 BOM 时 version = 1
⑥ 命名统一 — find → get（单 ID 查询），query → list（分页查询），publish 返回 BomSnapshot
⑦ 领域错误 — 使用 DomainError（定义见共享基础设施层），核心变体：NotFound / ConcurrentConflict / BusinessRule / InvalidStateTransition

Customer 客户

客户主档案，AR（应收账款）主体。支持公海/私海客户池、信用额度管控、多收货地址。owner_id=null 为公海客户（全员可见），有值为私海（归属业务员+上级可见）。下游模块：Sales、FMS、WMS

entity Customer {
  id: i64 PK
  code: String // 客户编码，自动生成
  name: String
  short_name: String?
  category: CustomerCategory
  status: CustomerStatus
  tax_number: String? // 统一社会信用代码
  invoice_title: String? // 开票抬头
  credit_limit: Decimal(20,4)? // 信用额度

  remark: String
  operator_id: i64 FK→users
  created_at: Timestamp
  updated_at: Timestamp
  deleted_at: Timestamp?
}

乐观锁与发布流程

BOM 更新和节点更新要求客户端传入 expected_version，冲突返回 ConcurrentConflict。publish 返回 BomSnapshot（不可变），语义更准确

// 乐观锁机制
// 新增 BOM 时 version = 1
// UPDATE boms SET ..., version = version + 1
// WHERE bom_id = $1 AND version = $2
// affected_rows == 0 → DomainError::ConcurrentConflict
// Handler 映射为 gRPC Aborted (ABORTED)
// 服务端不自动重试，客户端获取最新版本后重新提交

// 发布流程（BomCommandService.publish）
publish(ctx, db, bom_id) → BomSnapshot
  Draft → Published
  // version 自增
  // 自动生成 BomSnapshot(bom_id, version, ...)
  // 同事务内:
  // AuditLog.record(ctx, db, Publish, ...)
  // EventBus.publish(ctx, db, BomPublished{...})
  // 返回 BomSnapshot（不可变）

unpublish(ctx, db, bom_id) → Bom
  Published → Draft
  // 不删除快照

// 物料替换（InCallerTx + FOR UPDATE）
substitute_product(ctx, db,
  req: SubstituteReq)
  → SubstitutionResult {
    affected_boms, affected_nodes
  }

状态转换规则

ProductStatus
  Active ↔ Inactive — 双向，停用/重新启用
  Active → Obsolete — 作废，不可逆
  Inactive → Obsolete — 作废，不可逆
  Obsolete → * — 不允许任何转换

BomStatus
  Draft → Published — 发布，生成快照，version 自增
  Published → Draft — 取消发布，不删除快照

CustomerStatus
  Prospective → Active — 潜在客户转正
  Active ↔ Inactive — 停用/重新启用
  Active → Blacklisted — 拉黑，发布 CustomerBlacklisted 事件
  Blacklisted → * — 不允许自动恢复，需人工审批

SupplierStatus
  Prospective → Qualified — 准入通过
  Qualified ↔ Probation — 试用期降级/恢复
  Qualified → Disqualified — 取消资格
  * → Blacklisted — 拉黑，发布 SupplierBlacklisted 事件
  Blacklisted → * — 不允许自动恢复，需人工审批

工艺路线 & 劳务工序 — 业务校验规则

RoutingService.create
  Validation: steps 中所有 process_code 必须存在于 labor_process_dicts
  Validation: steps.step_order 同一 routing 内不可重复

RoutingService.delete
  软删除（SET deleted_at = NOW()）
  Validation: 无 bom_routings 绑定（否则返回 BusinessRule("路线已被 BOM 绑定")）
  Validation: MES 无工单引用该路线（查询 work_order_routings）
  Partial Unique Index: uk_routings_name ON routings(name) WHERE deleted_at IS NULL

RoutingService.find_matching_routing
  按 process_codes 集合精确匹配现有路线
  匹配逻辑：忽略 step_order 顺序，仅比较 code 集合
  返回 Option<RoutingDetail>（含 routing + steps）
  MES 快照时使用原始 step_order，匹配逻辑不改变顺序

RoutingService.set_bom_routing
  UPSERT bom_routings（product_code 唯一，每个产品只绑定一条路线）
  发布 BomRoutingChanged 事件

LaborProcessDictService.delete
  软删除（SET deleted_at = NOW()）
  Validation: 无 routing_steps 引用（WHERE process_code = ? AND deleted_at IS NULL）
  Validation: 无 bom_labor_processes 引用（WHERE process_code = ? AND deleted_at IS NULL）
  Partial Unique Index: uk_labor_process_dicts_code ON labor_process_dicts(code) WHERE deleted_at IS NULL

BomLaborProcessService.delete
  软删除（SET deleted_at = NOW()）

v7 设计改进 — AcquireChannel 枚举化（Issue #12 + #16）

变更摘要

变更类型 字段类型提升 String → 枚举
影响字段 Product.acquire_channel（从 ProductMeta JSONB 提升为独立列+索引）
新增枚举 AcquireChannel: SelfProduced(1) / Purchased(2) / Outsourced(3) / NonInventory(4) / Legacy(9)
设计依据 销售订单履行流程需要根据产品获取途径分流（自制→生产订单 / 外购→采购订单），String 类型无法保证类型安全和穷举匹配

迁移策略

  • DB 层：在 products 表新增 acquire_channel VARCHAR 列 + CHECK 约束 + B-tree 索引
  • 迁移：从 meta JSONB 中提取 acquire_channel 填充新列，无法识别的设为 Legacy(9)
  • Rust 层：AcquireChannel 枚举 + serde 字符串序列化，读写兼容现有数据
  • Legacy(9)：迁移时无法识别的历史数据标记，不阻塞业务流程，日志记录供数据清洗
  • ProductMeta JSONB 中移除 acquire_channel 字段（双写过渡期后清理）
  • 下游模块（sales/purchase/wms）通过 AcquireChannel 枚举进行穷举匹配，编译期保证无遗漏
