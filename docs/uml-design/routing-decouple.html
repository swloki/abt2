① 现状问题：为什么每个 BOM 都要配一条 routing？

根因：routing_steps 这张「工艺模板行」同时塞进了三个本应分离的维度，任一差异整条分叉。

工艺结构

"焊接 5 分钟、需 IPQC" — 无论焊哪个产品都成立，可共享。

产出品 product_id

"这道工序产出哪个中间品" — 属于具体 BOM 的中间品，不同产品不同。

计件价 unit_price

"这道工序计件工资多少" — 商务/产品特定，不同产品差异巨大。

关键反转：bom_labor_processes 表（迁移 013）本来就是 per-BOM 计件价表。 045 把 unit_price 复制到 routing_steps、063 又复制 product_id —— 是这两次复制制造了耦合，而不是 ABT 缺覆盖层。09-master-data 原本就设计了「三层工艺解耦」，是后来偏离了。

② 三家 ERP 怎么做（横向对比）

查了本地的 ERPNext / Odoo / Apache OFBiz 三家开源 ERP，对工艺路线的处理高度一致。

维度ABT 现状ERPNextOdooOFBiz
产出品归属 焊模板 routing_steps BOM.item，绝不在 operation mrp.bom.product_id，副产品走 byproduct 边表 WorkEffortGoodStandard
价格 / 工时 焊一行 unit_price+工时 彻底分离：价格归工作中心（hour_rate / costs_hour），工时归工序，运行时相乘

四条铁律（三家无一例外）

① 产出品不在 routing/operation

产出品归 BOM 或独立边表，绝不焊在工艺主数据上。

② 价格归工作中心，工时归工序

调价只改工作中心，不动工艺主数据。运行时 工时×费率。

③ 复用靠引用/快照

共享模板 + per-instance 覆盖分层，不靠字段堆叠。

④ copy-on-write 可接受

模板改了不自动回流到已建工单（快照断层），三家共识。
ABT 已有两个被低估的解耦基础设施：labor_process_dicts ≡ ERPNext Operation（全局工序库已存在）； work_center.costs_hour 已是 Odoo 式制费（overhead = work_hours × costs_hour 已正确计算）。
问题精确锁定在两个字段的搬家，无需大重构。
China 特色：三家都是工时制，ABT 的 unit_price（计件工资）是独有 — 既不能纯归工作中心也不能留模板，唯一正确归属是 per-BOM-per-step。

④ 目标设计（After）

把产出品 + 计件价从 routing_steps 下沉到 per-BOM 覆盖层，routing 回归纯工艺模板。

routing_steps（瘦身）

  • 保留：工艺结构（工序/工作中心/工时/委外/检验点）— 真正可共享
  • 降级：product_id / unit_price → 可空模板默认（RT000001 家族沿用）
  • 首期不删列，保守 additive

bom_routing_outputs（新增）

  • per-BOM-per-step 的产出品 + 计件价覆盖
  • ≡ Odoo bom.byproduct.operation_id / OFBiz 边表
  • 工序在 A 产品的产出/价格，不影响共享同一模板的 B/C/D

bom_labor_processes（保持 legacy）

  • 角色收敛为"无 routing 的独立人工工序"
  • 带 quantity 按数量计费，与覆盖层计费模型不同
  • 审计证实强行合并会丢信息 → 双轨

work_order_routings（快照）

  • 列不变，只改 load 的数据来源
  • 下游报工 / 领料 / 委外 零侵入
  • copy-on-write 断层（三家共识）

APR-A · 快照 product_id 回填（修现网 bug）

  • 纯数据 migration：按 work_orders.routing_id + step_no JOIN routing_steps 回填历史快照的 product_id
  • 仅回填 RT000001 同构家族（模板有 product_id）；RT000002 模板空 → 业务单独补齐
  • 不改任何代码路径 → 零回归，单独修复 OM 委外读不到产出品的现网 bug

BPR-B · 覆盖层 + 前端解耦

  • 建表：bom_routing_outputs + 新模块 master_data/bom_routing_output/ + Service trait
  • load 切换：load_routings_from_template 加 product_code 参数，读覆盖层回退模板（INSERT 列不变）
  • 前端：routing 编辑页删"产出品/计件价"两列 + 详情页按关联 BOM tab 填差异（彻底消灭 Issue #212 候选裁剪）
  • 复用入口：find_matching_routing 接入 Web 新建流程（命中复用+提示填覆盖）

护栏

step 关联键

对"有覆盖"的 routing 禁止破坏性 step 重排（参照已报工锁定先例），仅允许末尾 append

InProduction 锁

切换数据源期间，对在制工单禁止 reload（避免成本快照前后不一致）

多源截止

保守 additive 设硬性升迁截止，禁止长期向 routing_steps 写新 product_id/unit_price
